================================================================================
  GESTOR BIBLIOTECA — Manual de instalación y uso
  (gestor-biblioteca · github.com/Luix70/gestor-biblioteca)
================================================================================

Agente de catalogación automática de biblioteca personal. Ingiere libros y
revistas en formato EPUB, PDF o imágenes escaneadas, extrae y enriquece sus
metadatos (ISBN, título, autores, editorial, CDU…) usando APIs gratuitas y
IA (Gemini), y los persiste en MongoDB Atlas con portada, sidecars y árbol
de ficheros clasificado por CDU.


────────────────────────────────────────────────────────────────────────────────
1. REQUISITOS PREVIOS
────────────────────────────────────────────────────────────────────────────────

  · MongoDB Atlas          — base de datos en la nube (tier M0 gratuito sirve).
                             La colección "biblioteca" se valida con $jsonSchema;
                             créala antes o deja que la app la cree sola al
                             insertar el primer documento.
  · Gemini API Key         — console.cloud.google.com · "Generative Language API"
  · Google Books API Key   — console.cloud.google.com · "Books API"
  · Docker + docker-compose en el NAS (Synology: Compose v1, "docker-compose").
  · Node.js 18 en local si se quiere desarrollo o batería de tests.

2. ESTRUCTURA DE DIRECTORIOS EN EL NAS
────────────────────────────────────────────────────────────────────────────────

  /volume1/docker/
    GestorBiblioteca/          ← código de la app (bind-mount en el contenedor)
      .env                     ← secretos y overrides (NUNCA en git)

      ...

  /volume3/BIBLIOTECA DIGITAL/

    Reintentos/   ← fallos de infraestructura (red, BD); se reintentarán

3. PRIMERA INSTALACIÓN EN EL NAS
────────────────────────────────────────────────────────────────────────────────

  3.1  Conectar por SSH al NAS
       ssh usuario@ip-del-nas

  3.2  Crear directorios de datos
       mkdir -p "/volume3/BIBLIOTECA DIGITAL/Inbox"
       mkdir -p "/volume3/BIBLIOTECA DIGITAL/CDU"
       mkdir -p "/volume3/BIBLIOTECA DIGITAL/Cuarentena"
       mkdir -p "/volume3/BIBLIOTECA DIGITAL/Reintentos"

  3.3  Crear el directorio de la app y descargar el código
       mkdir -p /volume1/docker/GestorBiblioteca
       cd /volume1/docker/GestorBiblioteca
       wget -qO- https://github.com/Luix70/gestor-biblioteca/archive/refs/heads/main.tar.gz \
           | tar -xz --strip-components=1

       (Si el repo es privado, añade el token de GitHub:)
       wget --header="Authorization: token ghp_TU_TOKEN" -qO- \
           https://github.com/Luix70/gestor-biblioteca/archive/refs/heads/main.tar.gz \
           | tar -xz --strip-components=1

  3.4  Crear el fichero .env  (NO va a git; es la única fuente de secretos)
       cat > /volume1/docker/GestorBiblioteca/.env << 'EOF'
       MONGO_URI=mongodb+srv://USUARIO:PASSWORD@cluster.mongodb.net/?retryWrites=true
       MONGO_DB_NAME=Biblioteca
       GEMINI_API_KEY=tu_clave_gemini
       GOOGLE_BOOKS_API_KEY=tu_clave_google_books
       TZ=Europe/Madrid
       EOF
       chmod 600 /volume1/docker/GestorBiblioteca/.env

  3.5  Copiar el script de actualización fuera del directorio de la app
       cp /volume1/docker/GestorBiblioteca/actualizar-GestorBiblioteca.sh \
          /volume1/docker/actualizar-GestorBiblioteca.sh
       chmod +x /volume1/docker/actualizar-GestorBiblioteca.sh

  3.6  Construir e iniciar el contenedor
       cd /volume1/docker/GestorBiblioteca
       docker-compose up -d --build

  3.7  Comprobar que arrancó bien
       docker-compose logs -f --tail=50
       # Debes ver algo como:
       #   🚀 API REST de ingesta activa en el puerto 3000
       #   👁️  Vigilante observando: /app/Inbox
       #   🧹 Conformador activo: mantenimiento tras 300s de Inbox inactivo.

────────────────────────────────────────────────────────────────────────────────
5. ACTUALIZAR LA APP EN EL NAS
────────────────────────────────────────────────────────────────────────────────

  Desde el NAS (o vía SSH):

      sudo bash /volume1/docker/actualizar-GestorBiblioteca.sh

  El script:
    1. Descarga el tarball de la rama main de GitHub
    2. Para el contenedor y elimina el volumen de node_modules
       (esto evita que dependencias viejas "ensombrezcan" las nuevas)
    3. Sincroniza el código preservando .env, node_modules y los datos
    4. Reconstruye la imagen e inicia el contenedor
    5. Muestra los logs en tiempo real (Ctrl+C para salir; el contenedor sigue)

  Si el repo es privado, exporta el token antes:
      sudo GITHUB_TOKEN=ghp_TU_TOKEN bash /volume1/docker/actualizar-GestorBiblioteca.sh

  ROLLBACK: la etiqueta git "nas-estable" marca el último commit bueno
  desplegado. Para revertir:
      git reset --hard nas-estable && git push --force origin main
      sudo bash /volume1/docker/actualizar-GestorBiblioteca.sh

6. CÓMO CATALOGAR LIBROS (EL INBOX)
────────────────────────────────────────────────────────────────────────────────

  El vigilante del Inbox monitoriza continuamente la carpeta Inbox.
  Deposita ficheros en cualquiera de estas carpetas por SMB/SFTP y la app
  los catalogará automáticamente:

    /volume3/BIBLIOTECA DIGITAL/Inbox/

  Formatos soportados:
    · .epub               Libros electrónicos
    · .pdf                Libros/revistas PDF (con texto o escaneados)
    · .jpg .jpeg .png     Imágenes de páginas escaneadas
    · .webp .heic         Otros formatos de imagen
    · .mobi .azw .azw3    Kindle/Mobipocket: lector propio (EXTH → título/autor/
                          editorial/ISBN + portada embebida). El ISBN sirve de pivote.
                          Con DRM NO se puede leer → se OMITE (se deja en el Inbox).
    · .cbr .cbz .cb7      Cómics (primera imagen = portada; visor paginado bajo demanda)
    · .djvu               Libro escaneado: rasteriza páginas para la visión (ISBN/portada) y el visor
    · .chm                HTML compilado (manuales/libros): libchm-bin extrae título/ISBN/portada
    · .zip .rar .7z .iso  Se EXPANDEN a carpeta (drop) antes de catalogar (bsdtar; .iso = ISO9660)

  Papel vs digital (filtro «Soporte»):
    · El formato lo determina el FICHERO (su extensión), no lo que la IA vea en la
      página. Un PDF ESCANEADO se identifica por VISIÓN pero SIGUE SIENDO DIGITAL:
      su soporte es «pdf», se conserva el fichero y solo se guardan hasta 6 páginas
      frontales como vista previa. (Al subir un PDF por el panel se envía ÍNTEGRO;
      NUNCA se explota en páginas-JPG en el navegador —eso lo convertía en «papel»
      con decenas de imágenes y perdía el PDF—.)
    · «papel» se reserva a los ejemplares FÍSICOS: fotos/escaneos sueltos (.jpg/.png)
      de un libro sin fichero digital, las altas por ISBN sin fichero, y —por la
      política de páginas— los ESCANEOS FINOS (ver abajo). En ese último caso el PDF
      original SE CONSERVA en la carpeta junto a las páginas extraídas (salvaguarda).

  Clasificación por nº de PÁGINAS (solo PDF):
    · Escaneado con MENOS de 12 páginas → libro de «papel» (folleto/ejemplar físico
      fino); el PDF se conserva igual. Entre 12 y 20 → libro PDF digital.
    · PDF LEGIBLE (con texto) con MENOS de 20 páginas y SIN ISBN/CIP propio →
      «Artículo» (si trae DOI) o «Capítulo» (fragmento de un libro). Con ISBN/CIP
      propio es un librito real y queda como «Libro».
    · 20 páginas o más → libro o revista, según su ISBN/ISSN (como siempre). Una
      REVISTA ya detectada (ISSN/fecha) nunca se reclasifica por páginas.
    · Umbrales configurables: CLASIF_PAPEL_MAX_PAGINAS (12), CLASIF_CAPITULO_MAX_PAGINAS
      (20). Tipos de documento: Libro · Revista · Cómic · Artículo · Capítulo · Apuntes.

  Agrupación de imágenes:
    · Varias imágenes en la MISMA CARPETA dentro de Inbox se tratan como
      UN SOLO libro (páginas escaneadas de la misma obra).
    · Imágenes sueltas en la raíz del Inbox también se agrupan como un libro.
    · Cada EPUB o PDF es siempre una unidad independiente.

  Soltar una CARPETA = COLECCIÓN:
    · Una subcarpeta del Inbox con 2+ documentos (a cualquier profundidad) se
      cataloga como una colección con el NOMBRE de la carpeta; cada documento va
      bajo su propia CDU pero queda ligado (coleccion + nº de serie autoincremental
      o el del nombre de fichero). La carpeta NO se borra (buzón persistente): se
      puede seguir soltando ahí y se FUSIONA con la colección previa.
    · Una subcarpeta "Covers/" (o un Book.jpg junto al doc) aporta portadas
      candidatas (compiten por tamaño con la embebida/remota; "gana la más ancha").
    · 1 solo documento = NO es colección; los .txt/.url sueltos se descartan.

  Pipeline de catalogación (automático):
    1. Extracción de metadatos del fichero (título, ISBN, autor…)
    2. Enriquecimiento con APIs gratuitas: OpenLibrary, Google Books
    3. Si faltan datos: Gemini (visión IA) extrae lo que no se encontró
    4. Clasificación CDU (caché aprendida → API → IA)
    5. Guardado en MongoDB Atlas
    6. Copia de ficheros al árbol CDU + portada + sidecars (registro.json,
       registro.marc.xml)
    7. Eliminación del Inbox (solo si la copia se verificó íntegra)

  Resultado por fichero:
    · Éxito     → desaparece del Inbox, aparece en CDU/ con todos sus datos
    · Error de identificación → va a Cuarentena/ con un estado.json descriptivo
    · Error de red/BD        → va a Reintentos/ (el Conformador lo reintentará)
    · NO se puede procesar (DRM, audio, formato sin lector) → NO se borra ni va a
      Cuarentena: se DEJA en el Inbox intacto. El vigilante deposita un testigo
      «.noborrar» en la carpeta de PRIMER NIVEL del drop y el recolector de basura la
      respeta (no la disuelve ni recicla su contenido). Borra el «.noborrar» si quieres
      que se reintente. El AUDIO (audiolibros, aún sin tratamiento) es «conservable»:
      nunca se recicla. (Un fichero omitido no se reintenta en cada escaneo.)

7. EL PANEL DE CONTROL WEB (la app que usas a diario)
────────────────────────────────────────────────────────────────────────────────

  Además del Inbox por carpeta, la app sirve un PANEL WEB completo en la raíz
  (http://<nas>:3000/ — o vía proxy del NAS). Es una sola página (public/index.html)
  con todas las funciones. Pensado MÓVIL-FIRST e instalable como app (PWA).

  ACCESO Y SESIÓN
    · Login con usuario/contraseña: rol ADMIN (todo) o INVITADO (solo lectura;
      no cambia NADA en la base ni dispara IA, y por defecto no ve el contenido
      🔞 NSFW). La sesión usa un token firmado que SOBREVIVE a reinicios del
      contenedor y dura ~2 días (no hay que reloguear a cada rato).
    · Instalar como app (PWA): el navegador ofrece "Instalar"/"Añadir a inicio".
      Queda como icono propio ("Bibliotheca Ludoviciana").
    · 📌 FIJAR ORIENTACIÓN (solo móvil): botón en la barra superior que BLOQUEA la
      pantalla en la orientación ACTUAL (vertical u horizontal, la que haya al pulsarlo)
      para que no rote; se resalta (pri) mientras está activo, se recuerda entre sesiones
      y se re-aplica al cargar. Vía Screen Orientation API (ignora la autorrotación del
      sistema). En la PWA instalada (standalone) bloquea sin más; en pestaña normal Chrome
      EXIGE pantalla completa, así que si el bloqueo directo falla se entra en pantalla
      completa y se reintenta (al soltar el pin, o al salir de pantalla completa con
      Atrás/Esc, se libera). Solo aparece en dispositivos táctiles con soporte; degrada en
      silencio. Comparte mecánica (_fijarOrientacion/_soltarOrientacion) con el bloqueo
      automático del etiquetado NFC (si el pin está activo, cerrar un modal NFC no lo suelta).

  NAVEGACIÓN (menú ≡): Catálogo · Dashboard · Obras · Colecciones · Autores ·
  Editoriales · Ubicaciones · Inbox · Actividad · Cuarentena · Papelera.
  En móvil el botón Atrás del teléfono navega hacia atrás (doble-atrás = salir).

  VERSIÓN DESPLEGADA: el pie del menú muestra «v1.<serie>», donde la serie es el
  nº de commit (sube con cada despliegue) — así se ve de un vistazo qué build corre
  sin adivinar. El tooltip añade el commit/rama. La calcula el script de despliegue
  (cabecera Link de la API de GitHub) y la sirve GET /api/version (pública). Si el
  NAS no puede leer la serie, muestra el commit corto.

  ── CATÁLOGO (búsqueda + selección) ──────────────────────────────────────────
    · Caja de búsqueda (título, TÍTULO ORIGINAL, autor, editorial, ISBN, ISSN, DOI,
      nombre de archivo). El título original se pliega en la columna `titulo` del índice,
      así «War and Peace» encuentra «Guerra y paz» (requiere REINDEXAR tras desplegar:
      panel → Búsqueda → reindexar, o scripts/reindexar-busqueda.js; las altas/ediciones
      nuevas ya lo incluyen solas). Rápida e INSENSIBLE A ACENTOS gracias a un índice
      local SQLite-FTS ("matematicas" encuentra "Matemáticas"); si no está, cae a Mongo.
    · «🎯 Frase exacta» (casilla bajo la caja): modo ESTRICTO. Por defecto la búsqueda es
      LAXA (casa TODAS las palabras sueltas, en cualquier posición: «history of philosophy»
      encuentra también «Philosophy and the History of Science»). Con la casilla marcada solo
      salen los que tienen la FRASE EXACTA adyacente y en ese orden (FTS: «"history of
      philosophy"»; &estricto=1). Sigue siendo insensible a acentos/mayúsculas.
    · VARIANTES de nombre (autor/editorial): el FTS indexa solo el nombre PRINCIPAL; las
      grafías alternativas (otros alfabetos, formas castellanizadas, seudónimos) viven en el
      registro de autoridad y se resuelven por MONGO, uniendo sus documentos a los aciertos
      del FTS. Así «Жюль Верн» o «Julio Verne» encuentran a «Verne, Jules»; «Диккенс» a
      «Dickens, Charles». NO se indexan en el FTS A PROPÓSITO: engordarían la columna
      `autores` y el bm25 penaliza por LONGITUD de campo, hundiendo en el ranking a los
      autores con muchas variantes (comprobado: con las variantes dentro, «Verne» devolvía
      «Vernet, Juan» ANTES que Jules Verne). El nombre principal se LATINIZA en la ingesta
      (utils/transliterar.js: cirílico y griego) y la grafía original queda como variante;
      resolver-persona empareja por principal o variante (no duplica autores) y la fusión
      de autores conserva todas las grafías.
    · Panel de filtros PLEGABLE (desplegado en PC, colapsado en móvil). Al confirmar
      una búsqueda (Enter / «Ir» del teclado móvil) se CONTRAE solo para ver los
      resultados sin scroll; el botón «✕ Limpiar» vive en la CABECERA (a la derecha,
      visible aun con el panel plegado). Filtros:
        - Tipo (Libros/Revistas/Cómics)
        - Soporte (Papel/Digital)
        - Formato concreto: PDF · EPUB · MOBI/AZW · CBZ · CBR · CB7 · DjVu · 🔊 Audio ·
          🎬 Vídeo · Papel (casa contra el array `formatos`; audio=audiolibros,
          video=vídeos de colecciones/transmedia). Whitelist server: FORMATOS_FILTRO.
        - Ámbito y Estantería (ubicación física)
        - CDU (prefijo)
        - Estrellas (valoración, multi-selección; "sin valorar"). Dentro del mismo
          desplegable, 🔞 NSFW (admin), con TRES estados: SIN marcar = OCULTA lo
          NSFW (por defecto); MARCADA junto a otros filtros = lo INCLUYE también;
          MARCADA y sola (sin otro criterio) = SOLO NSFW.
        - Etiqueta NFC (admin): Todas / Con etiqueta (📶) / Sin etiqueta
      El ORDEN está en su PROPIO colapsable «↕️ Ordenar» (el panel de búsqueda era
      enorme): Relevancia/recientes (defecto) · Fecha de ingreso · Título · Autor ·
      Posición en la estantería · Posición en la obra · Posición en la colección · Nº de
      páginas (los sin nº de páginas caen SIEMPRE al final, en asc y desc), cada
      uno con un botón de SENTIDO ascendente/descendente (oculto en Relevancia). El Nº de
      páginas se muestra ademas en la etiqueta de cada tarjeta/fila («📕 Libro · pdf · 298p»).
      En MÓVIL, al ejecutar la búsqueda (Intro / «Buscar» del teclado) se COLAPSA el panel
      de filtros y se cierra el teclado para ver los resultados (el input lleva
      enterkeyhint="search" para que el teclado muestre «Buscar» y no «Siguiente»). Al ver
      UNA estantería se ordena por POSICIÓN física (ya reflejado en el selector, ver
      Ubicaciones · 📋). «Posición en la estantería» ordena por el orden FÍSICO combinado
      ÁMBITO → ESTANTERÍA → posición dentro de la estantería (orden_estanteria, NUMÉRICO:
      2 antes que 11); los sin ubicación/posición caen al final.
    · VISTA (botón en la cabecera): «▦ Iconos» (rejilla con portadas, 24/pág, por
      defecto) o «☰ Detalles» (una fila de texto por documento —título · autor · año ·
      ISBN · CDU + formatos—, 100/pág, para revisar muchos de un vistazo). Se recuerda.
    · Resultados como tarjetas con miniatura; PAGINACIÓN arriba y abajo (al pasar
      de página, el scroll sube al primer resultado). Controles reutilizables (mismos en
      Catálogo, miembros de colección/obra, Autores y Descubrir): ⏮ primera · ‹ anterior
      · SALTO a una página concreta (input numérico, Enter) · siguiente › · última ⏭.
      PAGINACIÓN ESCALABLE (api-panel · /catalogo): el $sort corre sobre una PROYECCIÓN
      LIGERA (_id + SOLO las claves de orden) y luego se traen los documentos completos de
      la página por _id (reordenados con un Map, porque $in no conserva el orden).
      MOTIVO: MongoDB ordena en MEMORIA (tope 32 MB) y en Atlas M0 `allowDiskUse` NO está
      permitido; el optimizador convierte $sort+$skip+$limit en un top-K cuyo buffer vale
      (skip+limit) DOCUMENTOS ENTEROS → a partir de la página ~410 (≈9.900 × 3,4 KB ≈ 33 MB)
      reventaba con «Sort exceeded memory limit». Solo pasaba con los $sort SIMPLES
      (fecha_ingreso/titulo), que se absorben en el cursor find(); los que llevan $addFields
      (posición/autor/obra/colección) ya podaban campos en la capa de agregación. Ordenando
      documentos diminutos el buffer cae ~60× (escala a 10× documentos) y además el orden por
      AUTOR se acelera (descarta el array del $lookup antes de ordenar). Índices de apoyo
      (setup-mongo): idx_fecha_ingreso (orden por defecto), idx_titulo_es (con la MISMA
      collation española que la consulta — si no, el índice no sirve al $sort) e idx_doi.
      ⚠ Hay que correr `node scripts/setup-mongo.js` para crearlos.
    · VALORACIÓN por estrellas (estilo Lightroom) y marca 🔞 NSFW desde cada tarjeta.
    · MODO SELECCIÓN (admin, patrón ergonómico y móvil, COMÚN A TODA LA APP — Catálogo,
      ficha de autor, Colecciones, Obras, ficha de colección/obra y Estanterías/
      Ubicaciones; helpers reutilizables montarSelDocs + attachGesto). INTERACCIÓN
      UNIFICADA de las tarjetas:
        · clic / toque simple = ABRIR la ficha (previsualizar) — o marcar si ya
          estamos en Modo selección;
        · DOBLE CLIC (PC) / PULSACIÓN LARGA (móvil) sobre una tarjeta = CONMUTAR el
          Modo selección (sin perder lo ya seleccionado).
      Sigue existiendo el BOTÓN que alterna «👁 Modo previsualización» ↔ «🖱 Modo
      selección» (por si acaso; hace lo mismo que el gesto). Cuando el Modo selección
      está ACTIVO se ve de un vistazo: CADA tarjeta muestra su CÍRCULO de selección
      (aro vacío = sin marcar, relleno ✓ = marcada; patrón iOS/Google Photos — la
      señal más clara, mires donde mires), la marcada se «hunde» un poco (escala 96%),
      fondo un punto más claro + PASTILLA inferior «🖱 Modo selección…». En las listas fuera del Catálogo la
      acción es «🔍 Mostrar en Catálogo» (lleva la selección al Catálogo, ya marcada).
      La selección NO se pierde al conmutar el modo. Con selección hay acciones: 👁 Ver · 📚 Colección · 📖 Obra · 📍
      Estantería · 🚫 Quitar de estantería · 📶 Etiquetar · 🧹 Conformar · ✨ Enriquecer ·
      🎯 A fondo (lee cada libro con la visión y aplica lo que aporte) · 🔀 Cambiar tipo ·
      🖼️ Portada común (asigna la MISMA imagen de portada a toda la selección: se añade como
      portada en cada uno; las imágenes que ya tuvieran se conservan en el carrusel —
      POST /documentos/portada-lote) · ♻️ Reprocesar (las
      MISMAS acciones de la ficha individual que proceden en lote —«medir» no—, aplicadas una
      a una con progreso) · 🗑 Eliminar (pide
      contraseña) · Limpiar. «🗂 Todos los resultados» selecciona todas las páginas.
      «👁 Mostrar selección» es un TOGGLE que restringe el Catálogo a SOLO los
      documentos seleccionados (para revisar la selección y, en Modo selección, quitar
      los que no quieras tocándolos). Los BOTONES de acción se pliegan en un PANEL
      COLAPSABLE «⚙️ Acciones (N)» (como «Buscar y filtrar»: recuerda su estado, en móvil
      arranca plegado) para no saturar la barra; la cuenta «N sel.» queda siempre visible.
    · 🔭 Descubrir (Fichero): busca en el volcado offline OL+BNE (58,7 M) libros
      que NO tienes, con enlaces para conseguirlos (Anna's/Z-Lib/Libgen/Gutenberg).

  ── FICHA DE UN DOCUMENTO ────────────────────────────────────────────────────
    Datos catalográficos; AUTOR y contribuyentes (traductor/ilustrador/…) "clicables"
    (abren la ficha del autor); identificadores (ISBN/ISSN) "clicables" (buscan todo
    lo que comparte ese identificador); "otras ediciones" (ISBN alternativos con rol);
    SINOPSIS; CDU y demás clasificaciones con su título + ⓘ (texto largo) + nº de
    documentos que comparten el código (clic = búsqueda filtrada); chip 📍 con la
    UBICACIÓN (clic = ver los libros de esa estantería; doble clic / pulsación larga
    (admin) = CAMBIAR la ubicación —acción frecuente— con un mini-editor).
    · MEDIOS DIGITALES → ESTANTERÍA VIRTUAL POR CDU (no tienen ubicación física): el
      chip muestra 📚 su CDU + descripción como «estante», y debajo 📂 el CDU
      inmediatamente superior como «ámbito» (821.111(73) → 821.111; 004.65 → 004.6).
      Clic en el estante = ver sus COMPAÑEROS (los demás digitales con esa MISMA CDU,
      exacta); clic en el ámbito = todo lo que cuelga de él (CDU por prefijo). En las
      tarjetas del Catálogo, un digital sin ubicación física muestra «CDU <código>».
    Acciones (admin):
      ✏️ Editar (y bloquear para que el Conformador no lo cambie). Incluye CDU · Dewey ·
         LCC · LCCN (estas tres son solo metadatos: a diferencia de la CDU NO re-alojan el
         fichero) · DOI (artículos) · ISBN/ISSN + otras ediciones. SOPORTE (papel↔digital):
         al pasar a «Digital», el editor busca el FICHERO ORIGINAL en la carpeta del
         documento (archivoOriginal) y, si lo encuentra (p. ej. un PDF que se catalogó por
         error como 'papel'), fija su formato real (pdf/epub/…) y su nombre → queda
         descargable y previsualizable; si no hay fichero, avisa (súbelo por Cuarentena). TAMAÑO en la ficha: si
         está MEDIDO con el tapete se muestra «ancho × alto cm»; si solo hay el DECLARADO
         por la fuente bibliográfica (campo `dimensiones`, casi siempre solo el alto:
         «24 cm») se muestra atenuado como «· declarado». AUTORES+COLABORADORES:
         editor de FILAS (nombre + rol: autor/traductor/ilustrador/editor/prologuista/
         anotador/compilador), añadir/quitar/corregir. Cada fila es UNA persona, así el
         nombre puede llevar comas («Touchard, Jean») sin partirse (antes el separador por
         comas lo rompía). Se resuelve con resolverPersona (insensible a mayús./acentos →
         no crea duplicados). El rol 'autor' va a autores[]; los demás a contribuciones[].
         En REVISTAS aparece además «Mes (ejemplar)» + «Nº de ejemplar»: definen la
         identidad del número (recomputan clave_numero: AAAA-MM → n<nº> → AAAA).
      🔀 Cambiar tipo — reclasifica a mano a Libro / Revista / Cómic (individual y en
         LOTE desde la selección del Catálogo). No mueve la carpeta (la Integridad/un
         reproceso re-alojan luego en libros/ o revistas/). Útil para series de libros con
         ISSN que la ingesta temprana marcó revista.
      El CARRUSEL de la ficha ya NO abre la imagen en una pestaña nueva al tocarla:
         la LUPA (🔍) de la esquina la amplía a pantalla completa (lightbox; se cierra
         con ✕ / clic fuera / Esc y navega con ‹ › o ←/→).
      🖼️ Imágenes — reordenar/añadir/borrar + editor en el navegador: rotar,
         recortar, corregir PERSPECTIVA (4 esquinas) con «✨ Auto bordes». Auto-bordes
         detecta la cubierta en 3 pasadas (todas locales, sin IA): tapete verde →
         SEGMENTACIÓN por el color del FONDO (libro sobre superficie uniforme, foto en
         ángulo, sin tapete) → líneas (Sobel/Hough). Ajusta las esquinas antes de aplicar.
         Al ➕ Añadir / 📷 Cámara, si la foto está sobre el TAPETE reglado se recorta+endereza
         +mide automáticamente; el interruptor «📐 Tapete» del modal lo DESACTIVA (adjunta la
         imagen TAL CUAL, sin tocarla) y recuerda la preferencia (localStorage img_tapete).
         🖹 «Del documento» extrae imágenes del propio fichero y las añade al carrusel —
         p. ej. para rescatar la foto del autor del interior. Cubre: PDF (páginas, pdf.js),
         EPUB (imágenes embebidas, JSZip), cbz/cbr/cb7/djvu (páginas del backend) y mobi/azw3
         (imágenes embebidas). Rejilla PEREZOSA (solo descarga la miniatura visible) + selección
         múltiple → carrusel a plena resolución. En MOBI/AZW3 sin imágenes (o a voluntad), «📄
         Página de texto» rasteriza el HTML del libro (SVG foreignObject→canvas) conservando
         estructura (encabezados/negrita/cursiva/párrafos) + serif; no la fuente embebida.
      📐 Medir — estima el tamaño físico (cm) sobre la alfombrilla reglada.
      🧹 Conformar · ✨ Enriquecer · ♻️ Reprocesar · 🗑 Eliminar (a la Papelera)
      📶 Grabar NFC (ver más abajo)
      ♻️ REPROCESAR ofrece DOS MODOS (se elige en el modal, con la contraseña):
         · CONSERVADOR (con sidecar, por defecto): escribe un sidecar .meta.json con los
           datos CURADOS (ubicación, colección + nº, obra + volumen, isbn, isbn_obra,
           valoración, nsfw, NFC) que SOBREVIVEN a la reingesta; el resto (título, autores,
           CDU, subtítulo, identificadores) se RE-DERIVA.
         · NUEVO DESDE CERO (sin sidecar): re-identifica TODO (nuevo _id; re-lee el CIP y
           recalcula ISBN/título/autor/colección…). Para arreglar un dato guardado ERRÓNEO
           (p. ej. un ISBN equivocado que el modo conservador volvería a imponer).
         En escaneos, las imágenes se re-empaquetan con prefijos 001_,002_… y la PORTADA
         primero, para conservar orden y portada.
    En «Datos catalográficos» el bloque de agrupación va en orden: Obra · ISBN obra ·
    Volumen nº · Título del volumen · Colección · Nº en colección (el nº de colección
    justo tras la colección).
    La PRIMERA fila es «ID (Mongo)»: el ObjectId del registro, COPIABLE (📋) y —para el
    admin— DRILLABLE: al pulsarlo abre un visor con el documento EXACTO de la base en JSON
    (GET /documentos/:id/crudo, solo admin), copiable entero y de SOLO LECTURA. A diferencia
    de la ficha, ahí NO se resuelven autores/editorial (salen como ObjectId en hex) ni se
    ocultan los campos internos (mantenimiento, mantenimiento_firma…): se ve lo que hay.
    El invitado ve el id y puede copiarlo, pero no abre el visor.
    DENTRO del JSON, cada ObjectId de 24 hex lleva su 📋 y —si por su CLAVE se sabe a qué
    colección apunta— es CLICABLE: `autores[]` y `contribuciones[].persona` → ficha de autor,
    `editorial` → ficha de editorial, `coleccion`/`obra` → su página. El `_id` y los hex sin
    clave conocida solo se copian; un hash de 64 hex no se confunde con un ObjectId.

  ── OBRAS (multivolumen) ─────────────────────────────────────────────────────
    La ficha de una obra lista sus TOMOS (presentes y los que faltan) y los «sin
    número». Admin: «🔢 Numerar tomos» abre un editor con un campo de nº por libro
    (vacío = sin número) + el TOTAL de tomos de la obra; al guardar reescribe el
    volumen_numero de cada libro y reconstruye el inventario. Anti-pérdida: dos
    tomos con el mismo número → el segundo queda «sin número» para revisión.
    (POST /obras/:id/numerar {numeros:{docId:nº|""}, total}.)  «⚙️ Orden automático»
    rellena la numeración por INDICIOS (nº ya asignado / «Vol./Tomo N» del
    título/OCR/nombre de archivo, arábigo o romano) y es editable antes de guardar.
    DETECCIÓN EN LA INGESTA (igual sea cual sea la vía de entrada — PDF/EPUB/imágenes/
    ISBN): además del drop de carpeta de tomos y del ISBN-con-rol de los créditos, el
    enriquecedor deduce el tomo del PROPIO TEXTO cuando el título/subtítulo/nombre de
    archivo trae «Vol./Tomo N» con el título de la obra delante (motor-enriquecimiento).
    Conservador: solo si no hay ya nº de volumen, y no en revistas ni cómics.
    CARPETA ÚNICA POR OBRA: todos los tomos viven JUNTOS en
    /CDU/<cdu>/obras/<isbn_obra | título>/vol-N/ (carpeta por isbn_obra si la obra lo
    tiene; si no, por su título). motor-catalogo IGUALA en todos los tomos el CDU, el
    isbn_obra y el obra_titulo al valor CANÓNICO del registro de obra, así que un tomo
    añadido DESPUÉS cae siempre en la misma carpeta (antes se derivaba de campos del
    tomo y podía caer aparte). Para consolidar lo ya catalogado (obras cuyos tomos
    quedaron dispersos): node scripts/consolidar-obras.js (dry-run; --ejecutar mueve las
    carpetas con verificación y actualiza ruta_base/portada/imágenes; BACKUP antes).

  ── ARTÍCULOS (el DOI es el pivote, como el ISBN lo es del libro) ────────────
    Un artículo es como un libro pero con o SIN identificador propio (igual que hay
    libros con y sin ISBN): incluye artículos de revista, capítulos sueltos y artículos
    escaneados. AUTO-DETECCIÓN por DOI: un DOI de revista (10.xxxx/… SIN ISBN incrustado)
    y SIN ISBN propio ⇒ `tipo_recurso:'articulo'` (discriminador). Guardas: un DOI de LIBRO
    Springer lleva el ISBN incrustado (…/978…) → sigue siendo libro; el ISBN del CUERPO de
    un artículo es de un LIBRO CITADO en las referencias → se DESCARTA (no es su identidad).
    El DOI se extrae del cuerpo + info-dict (Subject/Keywords) + nombre (lector-pdf·extraerDOI)
    y se guarda en `doi`. PIVOTE: CROSSREF (api.crossref.org, GRATIS y sin clave) resuelve el
    DOI → título REAL (sustituye el nombre del PDF), autores, revista de origen + ISSN, año,
    vol/nº/pp (bajo `articulo`) — es al artículo lo que el Fichero-por-ISBN al libro
    (buscador-crossref.js·buscarPorDOI; degrada en silencio; el DOI del dev tiene TLS
    interceptado → probar en el NAS o con NODE_TLS_REJECT_UNAUTHORIZED=0). Ficha: filas DOI
    (drillable a doi.org, copiable), Revista y Cita. Los artículos SIN DOI (capítulos,
    escaneos) se marcan a mano con «🔀 Cambiar tipo» → Artículo (como un libro sin ISBN).
    ENLACE A LA REVISTA (motor-catalogo 2f): el artículo se cuelga de la CABECERA de su
    revista (colección tipo:'revista', pivote ISSN — la misma donde caen los NÚMEROS de esa
    revista) vía resolverCabecera por `revista`+`issn`; el ISSN pasa a la cabecera (se retira
    del doc, como en 2e) y la revista queda como su Colección (drillable). Identidad/DEDUP del
    artículo = su DOI (como el ISBN del libro). DOI editable en la ficha (campo «DOI (artículos)»)
    y BUSCABLE en el Catálogo (pega el DOI o su URL doi.org/).

  ── EDITAR COLECCIÓN / OBRA (admin, en su ficha) ─────────────────────────────
    «✏️ Editar datos» abre un modal (MISMA interfaz para ambas): presentación/sinopsis,
    editorial (por nombre → check-then-create), CDU (cdu_manual) y FECHAS de publicación
    inicio/fin (año; fin vacío = «actualidad» — p. ej. una revista de 1920-1960 o de 1980
    a hoy). La colección lleva además su ISSN (autoridad de la cabecera/serie); la obra su
    ISBN de obra + total de tomos. Identificadores validados por checksum (los malos se
    descartan con aviso). El rango de años se muestra en la cabecera. No mueve ficheros
    (un cambio de CDU aquí es de metadatos; cada miembro conserva el suyo).
    (POST /colecciones/:id/editar · POST /obras/:id/editar. editar-grupos.js.)

  ── COLECCIONES TRANSMEDIA (estructura de carpetas = una colección) ──────────
    Una carpeta con material heterogéneo (p. ej. «Oxford Bookworms Library»: PDFs de
    lectura + tests/ejercicios/soluciones + AUDIOS + portadas, en «Stage N/…») se cataloga
    como UNA colección tipo:'transmedia' cuya estructura de disco se CONSERVA VERBATIM,
    colgando de UNA sola rama CDU (…/<cdu>/transmedia/<colección>/). NO se reorganiza nada.
    Principio: AÑADIR siempre, BORRAR nunca, VERIFICAR la copia íntegra ANTES de reciclar
    el origen (src/utils/transmedia.js · ingestarTransmedia; copiarVerificado compara nº
    de ficheros y bytes; si el origen aún se copiaba, falla sin tocar el origen y reintenta).
    · DETECCIÓN (vigilante): es transmedia si trae marcador «.transmedia», o CONTENIDO
      INTERACTIVO (.exe/.swf/bundle .app/autorun.inf — CD-ROM), o AUDIO CON ESTRUCTURA (≥2 PDFs
      de lectura o carpetas «Stage N»). El audio SUELTO sin estructura NO es transmedia → es
      audiolibro. El vigilante ESPERA a que la carpeta esté ESTABLE (nº ficheros+bytes sin
      cambios durante CARPETA_ESTABLE_MS) antes de operar — una copia de 19 GB tarda; una vez
      vista como transmedia lo sigue siendo (pegajoso), no cae a la vía normal.
    · CONTENIDO INTERACTIVO (CD-ROM con .exe/.swf/.app): se COPIA VERBATIM y se PRESERVA íntegro
      (no se cataloga —no se ve en el visor— pero NO se omite ni se altera ningún fichero: romper
      la estructura lo inutilizaría). Si la colección es SOLO interactiva (0 PDF/audio) igualmente
      se copia y se crea la colección (0 miembros); nada se pierde.
    · MIEMBROS PLANOS (sin IA, 100% estructural): un documento por PDF, con nivel (Stage),
      unidad (el libro) y rol_material (lectura / test / ejercicios / solucionario / glosario
      / guia). Autor/título salen del nombre de la carpeta de unidad («S0 Autor - Título»).
      Dedup por hash: PDFs de igual hash → UN doc (los ficheros permanecen, no se borra ninguno).
    · AUDIOLIBRO = unidad con audio y SIN PDF de lectura → doc naturaleza:'audiolibro',
      formatos:['audio']. Una lectura CON audio se queda como libro con formatos:['pdf','audio']
      (el thumbnail avisa) + su playlist; los audios (audios[]) se reproducen desde la ficha
      (visor PDF + playlist si además hay PDF).
    · PORTADAS: la lectura usa la cover.jpg de su unidad; los materiales/guías reciben su
      PROPIA 1ª página rasterizada (poppler) en «<colección>/.portadas/» (oculta) → ni
      portadas repetidas ni docs con icono genérico. Esa carpeta empieza por punto, así que
      /recursos se sirve con dotfiles:'allow' (si no, express.static la 404).
    · ruta_fija:true en la colección y en cada miembro (+ marcador «.ruta_fija» en la raíz):
      Integridad/Conformador NO podan, mueven ni deduplican ese subárbol.
    · FICHA DE COLECCIÓN: barra de filtros por NIVEL (Stage) y por MATERIAL (rol), con
      contador «N de total»; filtra en cliente (muestra/oculta) sin recargar.
    · CDU e IDIOMA se DEDUCEN (editables luego en «✏️ Editar datos»).
    · CLI: node scripts/ingestar-transmedia.js <carpeta>           (dry-run: imprime el plan)
           node scripts/ingestar-transmedia.js <carpeta> --ejecutar (cataloga)
      Reparar una colección ya catalogada antes de estos fixes (fecha_ingreso ausente →
      no salían en el Catálogo; portadas repetidas; sin indexar en la Búsqueda):
           node scripts/reparar-transmedia.js [--ejecutar]          (dry-run por defecto)

  ── AUDIOLIBROS PUROS (carpeta de audio = 1 documento) ───────────────────────
    Una carpeta cuyo contenido es AUDIO (sin PDFs de lectura estructurados) se cataloga como
    UN documento `naturaleza:'audiolibro'` (NO una colección): todas las pistas como PLAYLIST
    y TODAS las imágenes que la acompañan (portada/contraportada/libreto) al CARRUSEL — nada se
    pierde. Es el «transmedia sin PDF de lectura». Estructura verbatim (CDs, libreto) con
    ruta_fija; destino …/<cdu>/audiolibros/<carpeta>/. (src/utils/audiolibro.js)
    · IDENTIFICACIÓN barata, sin IA, por orden:
      1) TAGS de audio (music-metadata; src/utils/lector-audio.js): `album`→título (quitando
         « CDN»), `artist`/`albumartist`→autor, `year`→año, `comment`→narrador («read by…»).
         REGLA autor: si el `artist` es CONSTANTE en todas las pistas = autor (Borges, Plutarch,
         Fournier); si VARÍA = son narradores (p. ej. «From Shakespeare With Love», un actor por
         soneto) → NO se inventa autor (queda sin autor + narradores listados). Nunca autor falso.
      2) NOMBRE de carpeta (limpiando basura de ripeo: «[Livre audio]», «[MP3]», «[192kbps]»,
         «(CD MP3)», «.partN») → rellena autor/título/año que falten.
      3) CÓDIGO DE BARRAS de la contra/portada (zxing local, sin IA; leerISBNdeImagenes) → EAN/ISBN
         → PIVOTE al Fichero (OL+BNE, buscarEnFicheroLocal): rescata autor/editorial/idioma/CDU/
         sinopsis/dewey/lcc de la edición — CLAVE en audiolibros CORALES sin autor de ID3 (sonetos).
         CONSERVADOR: ID3/carpeta MANDA y solo se rellenan huecos, salvo CDU e IDIOMA, donde el dato
         AUTORITATIVO del Fichero gana a la deducción (género/acentos). El ISBN se guarda en el doc.
    · CDU deducida del género (poetry→82-1, biography→929, else 82) e IDIOMA por heurística de
      acentos/palabras — ambos EDITABLES. La carátula EMBEBIDA en el audio se extrae a
      «.portadas/» si no hay imágenes sueltas. Si además trae el texto en PDF → formatos
      ['audio','pdf'] y se abre en el visor + la playlist.
    · Detección de unidades: audio DIRECTO en la carpeta = ella es 1 audiolibro; audio solo en
      subcarpetas = cada subcarpeta inmediata = 1 audiolibro (contenedor). EXCEPCIÓN: si TODAS las
      subcarpetas con audio son DISCOS de la misma obra («CD1/CD2/CD3», «Título CD N», «Disco 2»…,
      misma base) → es UN solo audiolibro repartido en CDs (no se parte; si no, CD2/CD3 caerían en
      el anti-duplicados por título). Anti-duplicados por título.
    · ANTI-BUCLE: un audiolibro «ya catalogado» (o una colección transmedia «ya existe») devuelve
      `permanente:true`; el vigilante lo añade a `omitidasDefinitivas` y NO lo reintenta (si no, la
      carpeta —conservada en el Inbox con .noborrar— se reprocesaría en cada escaneo).
    · VIGILANTE (cableado): un drop en el Inbox se enruta así — TRANSMEDIA si trae marcador
      `.transmedia`, contenido interactivo (.exe/.swf/.app) o audio CON estructura (≥2 PDFs de
      lectura o carpetas «Stage N»); si no, AUDIOLIBRO si hay audio suelto (≤1 PDF, sin
      estructura); si no, pipeline normal. Detección PEGAJOSA (audiolibroVistas), como transmedia.
    · CLI equivalente: node scripts/ingestar-audiolibro.js "<carpeta>"            (dry-run)
                       node scripts/ingestar-audiolibro.js "<carpeta>" --ejecutar  (cataloga)
    · DESCARGAS (ficha): «⬇ Playlist» baja todas las pistas en un ZIP; «⬇ Todo (ZIP)» baja TODA
      la carpeta ruta_base (audio + imágenes + extras). Endpoint PÚBLICO en streaming (bsdtar, sin
      RAM) GET /api/descargar/:id?que=audio|todo — vale para cualquier documento (no solo audio).
    · PLAYLIST (ficha): cada pista guarda `grupo` (disco: CD1/CD2… vía etiquetaDisco) y `duracion`.
      El reproductor muestra la DURACIÓN por pista y, si hay ≥2 discos, un SELECTOR «Todo/CD1/CD2…»
      que filtra la lista (la reproducción sigue el orden completo). Admin «🎚️ Reordenar»: ordena
      por Nº de pista/Título/Duración (asc/desc) y/o mueve a mano (↑↓/arrastrar, como estanterías);
      se PERSISTE con POST /documentos/:id/audios/orden (reordenarAudios, espejo de reordenarImagenes).
    · CONTROLES DE REPRODUCCIÓN (ficha, si hay ≥2 pistas, además del <audio> nativo): navegación
      ⏮ primera · ⏪ anterior · ⏩ siguiente · ⏭ última; y tres modos que PERSISTEN en el navegador
      (localStorage): «▶️ Continuo» (al acabar una pista pasa a la siguiente vs. detenerse al final de
      cada una), «🔁 Repetir» cíclico (ninguna → 🔂 pista → 🔁 lista) y «🔀 Aleatorio» (baraja el orden
      conservando la pista en curso). «Repetir lista» implica seguir aunque «Continuo» esté apagado.

  ── COLECCIÓN DE AUDIOLIBROS (varios audiolibros = una colección) ────────────
    Una carpeta con VARIOS audiolibros (p. ej. «Joseph Campbell» con muchos títulos) se cataloga
    como UNA colección (tipo:'audiolibros') cuyos miembros son documentos (src/utils/coleccion-
    audiolibros.js · ingestarColeccionAudiolibros): un AUDIOLIBRO por libro (playlist agrupada por
    apartado/disco), un doc por PDF, y un doc por VÍDEO (naturaleza:'video', descargable). Copia
    verbatim + ruta_fija. CLI: node scripts/ingestar-coleccion-audiolibros.js "<carpeta>" [--ejecutar].
    · DETECCIÓN RECURSIVA: una colección = la carpeta cuyos hijos son LIBROS (audio directo o
      subcarpetas que son TODAS partes disco/sección: «CD2», «[Disc 1]», «1_Origins…»); si los hijos
      son grupos (autores), se recurre → funciona sueltes un autor o varios.
    · CARPETA PLANA con VARIAS obras en el nombre («Autor -- Obra -- Disc N of M», caso Feynman) →
      colección con UN audiolibro por OBRA (por claveObra, colapsando «Six Easy Pieces, Lecture 1..6»
      en «Six Easy Pieces» con las lecciones como apartados). Un solo trabajo en la carpeta = 1 libro.
    · TÍTULO del libro = NOMBRE DE CARPETA (fiable; el álbum ID3 en colecciones viene por disco o mal
      etiquetado). TÍTULO de pista = el MÁS RICO entre ID3 y nombre de fichero (mejorTituloPista, en
      audiolibro.js): descarta los genéricos («Track 05», «Disc 01 of 10»), y si uno contiene al otro
      gana el superset, si no el más largo. ISBN del nombre de carpeta (…-9781565117358) → Fichero.
      Duplicados por título se CONSERVAN (anti-pérdida, no se fusionan).
    · NADA INVISIBLE: los vídeos se catalogan (naturaleza:'video'); la ficha los previsualiza con un
      <video> nativo si el navegador los decodifica (mp4/webm/ogv/m4v/mov) y ofrece descarga para el
      resto (.avi/.mkv/.wmv…). Lo demás no catalogable se lista en un MANIFIESTO «_contenido.txt» en la
      raíz de la colección. Todo se copia al árbol CDU (ruta_fija) — NUNCA va a la Papelera (a la
      Papelera va solo el ORIGEN del Inbox, un duplicado), así que vaciarla no pierde nada.
      (formatos +'video' en el $jsonSchema → correr setup-mongo tras desplegar.)
    · VIGILANTE (CABLEADO, enrutado por PESO): 1) TRANSMEDIA FUERTE (marcador .transmedia /
      interactivo / «Stage N», esTransmediaFuerte) → transmedia; 2) COLECCIÓN de audiolibros
      (esColeccionAudiolibros: ≥2 libros/obras, sin leer ID3) → aquí; 3) transmedia DÉBIL (audio +
      ≥2 PDFs sin estructura, esCarpetaTransmedia) → transmedia; 4) audiolibro suelto; 5) normal. Así
      el audio DOMINANTE con PDFs de guía (Campbell) va a colección, y las lecturas graduadas con
      audio (Oxford, «Stage») a transmedia. Pegajoso (colAudioVistas); duplicado → permanente → no se
      reintenta (anti-bucle).

  ── COLECCIONES · NUMERACIÓN (series de libros) ──────────────────────────────
    El nº de un libro dentro de una colección puede ser EDITORIAL (leído del ISBN/datos
    o del nombre — PREVALECE) o AUTOMÁTICO (asignado por el sistema — cede). Se distingue
    con coleccion_numero_auto. REGLA anti-conflicto en la INGESTA: si entra un libro con
    su nº editorial y ese nº ya lo ocupaba uno AUTO, el auto se renumera a un hueco libre
    y el entrante conserva el suyo → nunca hay conflicto (motor-catalogo). Las colecciones
    pueden tener libros SIN número (no todas se numeran). En la ficha de una serie de
    libros, admin: «🔢 Numerar» abre un editor (nº por libro, badge editorial/auto) con
    «⚙️ Orden automático» que respeta SIEMPRE los editoriales y —según el interruptor
    «rehacer también los auto»— conserva o rehace los automáticos, rellenando huecos y
    continuando desde el más alto; editar un nº a mano lo vuelve editorial.
    (POST /colecciones/:id/numerar {numeros:{docId:nº|""}, auto:{docId:true}}.)
    · Los miembros se muestran ORDENADOS por Nº de colección NUMÉRICO (1,2,…,11, no
      1,11,2 — coleccion_numero es string, se ordena con $convert). Igual el orden
      «posición en la colección/obra» del Catálogo.
    · RENUMERAR DIRECTO (admin): el chip del nº de cada libro/tomo es un botón (✏️) →
      mini-modal para cambiar ese nº al vuelo, sin abrir el editor completo.
    · SELECCIÓN: en modo selección hay «☑ Todos / ☐ Ninguno» (marca/desmarca toda la
      página). «🔍 Mostrar en Catálogo» de la selección ORDENA por Nº de colección
      (o de tomo en obras) — numérico y ascendente, reflejado en el selector de orden.
    · Lo mismo aplica a las OBRAS (tomos): botón «☑ Todos», renumerar tomo directo,
      y «Mostrar en Catálogo» ordenado por Nº de tomo.
    · 📷 NUMERAR POR LOMOS (admin, series de libros): foto(s) de los CANTOS alineados
      (de la CÁMARA o ARCHIVOS existentes) → el NAVEGADOR SEGMENTA cada lomo SIN IA
      (perfil de proyección: detecta los SURCOS entre libros; orientación AUTO por foto,
      así una en horizontal y otra en vertical se resuelven cada una) → filmstrip donde
      el admin AFINA (✂️ dividir un recorte que junte dos, ✕ quitar lo que no sea un lomo,
      ↻ cambiar orientación) → la VISIÓN lee el título y el nº de cada recorte YA AISLADO
      (mucho más preciso que pedir bbox; forzada a Gemini como los códigos de barras, en
      lotes de 10) → EMPAREJA con los libros por parecido de título (F1 sobre el título
      leído) → revisión (libro/nº editables + «adjuntar lomo») → renumera (nº del lomo =
      editorial) y adjunta los recortes. Si la segmentación no encuentra lomos (sin
      alfombrilla), reserva: envía las fotos completas y la IA da los bbox.
      (POST /colecciones/:id/lomos {recortados:1, imagenes:[dataURL]} — recortes ya
      aislados — ó {imagenes:[dataURL]} — fotos completas → propuesta; el cliente aplica
      con /numerar + /documentos/:id/imagenes/anadir. Segmentación: segmentarLomos() en
      public/app.js; lectura+emparejado: src/utils/lector-lomos.js.)

  ── INBOX (ALTA MANUAL desde el panel) ───────────────────────────────────────
    Organizado en fichas PLEGABLES (móvil-first): «📁 Subir ficheros» es la
    primera y siempre visible; «🏷️ Datos de esta alta» (los que VIAJAN) está
    colapsada; «⚙️ Modos y escaneo» (los interruptores) desplegada; «📷 Cámara» y
    «📶 NFC» colapsadas.
    Subir/arrastrar EPUB/PDF/imágenes/cómic/.zip/.chm/.iso; HACER FOTO con la
    cámara; IMPORTAR de otra app; o COMPARTIR a la PWA (Adobe Scan → Compartir →
    Bibliotheca). Antes de subir puedes fijar ISBN/colección/obra/ubicación que
    "viajan" con el alta. Un PDF se sube ÍNTEGRO (el servidor lo clasifica y
    conserva); solo se lee su código de barras de las primeras páginas como pista
    de ISBN, sin gastar IA.
    PROGRESO DE SUBIDA: la barra de estado muestra el % subido y, al 100%, «Subida
    completa · el servidor está catalogando…» (los ficheros grandes tardan un poco;
    ya no hay esperas «en silencio»). Si la ingesta se atasca (fichero enorme/pesado,
    INGESTA_TIMEOUT_MS = 8 min) o falla, el documento va a Cuarentena CON UN AVISO
    en la lista del Inbox (nunca desaparece sin más). Interruptores PERSISTENTES
    (ficha «Modos y escaneo»):
      🚥 Supervisado — tras cada alta abre la ficha para revisar/corregir; con
         "Cancelar" DESCARTA el alta (no se queda en la base). Si está apagado,
         AUTOPILOTO: encola y procesa en segundo plano (no bloquea; sigues subiendo).
      📤 Procesar lo compartido al instante (si no, queda EN ESPERA).
      📐 Tapete — la foto es un libro sobre la alfombrilla reglada (ver abajo).
      🖼️ Elegir portada — con varias fotos de un libro, antes de enviar muestra
         las miniaturas para confirmar/cambiar cuál es la portada (la del código
         de barras se asume contraportada).
    Cada trabajo muestra sus mensajes (ISBN leído, tapete, notas de visión…)
    DEBAJO, en la lista del Inbox.
    CÁMARA EN VIVO (Inbox → 📷): visor a pantalla completa con MULTIDISPARO. Trae:
      · DOS BOTONES FLOTANTES (caen a mano; un TOQUE ejecuta, MANTÉN+ARRASTRA para
        recolocar — posición persistente; misma mecánica, fabArrastrable): el de DISPARO
        (📸, círculo blanco) y el de CATALOGAR (embudo verde con insignia del nº de fotos,
        aparece con la 1.ª foto). Ambas acciones están también en la barra inferior.
      · FEEDBACK de captura: sonido de obturador + VIBRACIÓN + flash blanco + píldora
        «📸 Foto N» sobre el vídeo (el toast normal quedaría por debajo de la cámara por
        z-index). Tira de MINIATURAS para borrar fotos (✕) antes de procesar.
      · 📐 Recuadro on/off: superpone el cuadro del libro y la MEDIDA del tapete en vivo
        (verde con medida, ámbar midiendo); apágalo para fotos que no son portada. La
        detección corre a 720 px (igual que la foto fija) para que una calibración algo
        desajustada no lo haga «desaparecer» aunque en foto sí funcione.
      · ZOOM por FACTORES NATIVOS (botones mín·2·3·5·10×·máx según el dispositivo — evita
        la interpolación digital), persistente. 🔦 linterna si el aparato la tiene.
      · «✅ Catalogar» envía las fotos como un libro y SIGUE filmando (encadena libros).
        Con «Elegir portada» activo, el selector sale AL PULSAR Catalogar (sobre la cámara)
        y se ESPERA la elección antes de la siguiente tanda; las tandas NO se solapan (antes
        se lanzaban en paralelo y solo se catalogaba la última). Cancelar conserva las fotos.

  ── INGRESO POR ISBN (lector de código de barras o teclado) ──────────────────
    Escanea o teclea un ISBN y pulsa Enter/Buscar. Recupera los datos del FICHERO
    local (offline) y, si le faltan huecos (sinopsis/colección/CDU/portada) o no
    está, los COMPLETA con APIs gratuitas online, SIN pisar lo del Fichero.
      · ISBN mal formado (no 10/13 dígitos) o inválido: AVISO emergente (Toast) y
        se LIMPIA el campo al momento, para que no colisione con la próxima lectura.
      · Mientras busca, el campo se BLOQUEA (no encadena pulsaciones del escáner).
      · Los RESULTADOS salen en una ventana MODAL con las portadas grandes. Cada
        dato se colorea según su PROCEDENCIA (cuánto fiarse): negro = Fichero o
        escrito a mano (autoritativo) · azul = APIs gratuitas · rojo = IA/de pago.
        Al editar un dato a mano pasa a negro. Se elige/sube/conforma la portada y
        se pulsa «Crear» (rápido) o «Completar y crear» (enriquece con APIs+CDU).
      · Cualquier fallo del alta se avisa por Toast (no pasa desapercibido) y el
        modal sigue abierto para corregir.
      · Tras crear NO se salta a la ficha (interrumpía el flujo): el foco vuelve al
        campo ISBN para leer el siguiente; queda un enlace «ver ficha» por si acaso.
      · Auto-alta (interruptor + deslizador 1–10 s): tras esa inactividad da de
        alta con la portada elegida, limpia el ISBN y re-enfoca el campo para leer
        el siguiente — lectura en cadena con lector Bluetooth, sin tocar el ratón.
      · 📋 Alta por LOTE: pega una lista de ISBN (o del portapapeles, o sube un
        .txt); se buscan TODOS y se revisan en un CARRUSEL (un libro por pantalla)
        con la portada grande (se elige entre las candidatas) y TODOS los datos
        EDITABLES a mano (coloreados por procedencia). Casilla «Incluir» por libro;
        «Enviar seleccionados» los crea en bloque; los que fallen quedan en la lista.
        Campo «Colección para los que no la traigan»: se aplica SOLO a los que no
        traigan colección propia (útil para una serie entera, p. ej. Alianza Cien).
      · COLECCIONES por VARIANTE de grafía: al catalogar, una serie se asigna a la
        que YA existe aunque esté escrita distinta (orden, puntuación, conectores:
        «Cátedra, Letras Universales» ≡ «Letras Universales Cátedra»), casando por
        una CLAVE canónica (tokens significativos ordenados). Para unificar las YA
        duplicadas y rellenar esa clave en las existentes (necesario para que el
        emparejamiento funcione hacia atrás):
          node scripts/consolidar-colecciones.js            (cuenta, no escribe)
          node scripts/consolidar-colecciones.js --ejecutar (unifica; revisa antes)

  ── TAPETE (medir y recortar con la alfombrilla reglada) ─────────────────────
    Foto del libro sobre una alfombrilla de corte CUADRICULADA (1 cm). Con el
    interruptor 📐 Tapete activo, al subir: MIDE el libro (lee el paso de la
    rejilla, robusto al giro del tapete y del libro), RECORTA y ENDEREZA la imagen
    (quita el tapete) y guarda las DIMENSIONES en la ficha. Todo en el navegador,
    SIN IA. Botones:
      🎨 Calibrar tapete / 📷 Foto — fija el color de sesión del tapete a partir de
         una foto del tapete VACÍO: «🎨 Calibrar» la ELIGE de la galería y «📷 Foto»
         la TOMA al momento con la cámara (móvil). Sirve para CUALQUIER color (verde,
         teal, rosa, negro…). Rehazlo si cambia la luz (nublado/soleado).
      🔍 Probar — sobre una foto, pinta en ROJO lo que considera tapete y dibuja
         el contorno del libro detectado + la medida (para verificar antes de subir).

  ── NFC (etiquetas, Android + Chrome) ────────────────────────────────────────
    · Etiqueta de LIBRO (📶 en la ficha): graba un enlace que abre la ficha al
      acercar el móvil, MÁS datos OFFLINE en la propia etiqueta (ubicación, título,
      autor, editorial, colección, CDU, ISBN) para recolocarlo SIN conexión.
    · EX-LIBRIS CONFIGURABLE + DEVOLVER-SI-SE-PIERDE: la cartela «Este libro pertenece
      a…» es configurable desde el .env (NOMBRE_BIBLIOTECA / NOMBRE_BIBLIOTECARIO /
      EMAIL / TELEFONO), vía el endpoint PÚBLICO /api/exlibris (cacheado en el móvil
      para la lectura offline). Se muestra en 3 líneas (biblioteca · propietario ·
      «email — teléfono») y se graba en la etiqueta. Si alguien que NO tiene sesión
      escanea la etiqueta de un libro (típicamente quien lo ENCONTRÓ), en vez del login
      ve una TARJETA PÚBLICA con la propiedad + contacto para devolverlo (datos offline
      de la etiqueta, ?o=), con un botón «Entrar» para el propietario.
    · COMPARTIR (🔗, con QR + Web Share nativo → WhatsApp/Telegram/…): genera un enlace
      PÚBLICO firmado (?s=<token>) a un DOCUMENTO (ficha + descarga si es digital) o a un
      GRUPO —una COLECCIÓN u OBRA— con la lista de todos sus documentos y descarga
      individual. La vista pública es autónoma (sin login ni acceso al resto del catálogo).
      Botón en la ficha del documento y en las páginas de obra/colección. Backend:
      firmarCompartir/validarCompartir (token con `tipo` doc|coleccion|obra) +
      /compartido/:token (público). AVISO: el token NO caduca (enlace permanente hasta que
      cambie el secreto) y los ficheros bajo /recursos ya son públicos por ruta.
    · Etiqueta de ESTANTERÍA (desde Ubicaciones): al acercar el móvil lista los
      libros que contiene.
    · Leer (📲 en el Inbox): con conexión abre la ficha; SIN conexión muestra una
      tarjeta con la UBICACIÓN destacada (leída de la etiqueta) para recolocar.
    · BLOQUEO DE ROTACIÓN: mientras se graba (libro, estantería o lote) se BLOQUEA la
      rotación de pantalla para que no gire al manipular el móvil sobre la etiqueta —
      vía Screen Orientation API, IGNORA la autorrotación del sistema. En la PWA
      instalada basta; en pestaña normal Chrome exige pantalla completa (se entra y se
      sale solo). Se libera al cerrar el modal (cerrarCmp). Degrada en silencio si el
      navegador no lo permite (nunca rompe el etiquetado).
    · Al grabar con éxito se VINCULA la etiqueta al documento (su UID único + fecha
      + URL). Así puedes filtrar "con/sin etiqueta" en la Búsqueda; los que ya la
      tienen muestran un icono 📶 (cuarto de círculo en la esquina superior DERECHA)
      sobre TODAS sus miniaturas: búsqueda, carrusel de la ficha, tomos de obra y
      miembros de colección.
    · SOBRESCRIBIR: si vas a grabar sobre una etiqueta que YA contenía datos de otro
      libro, AVISA y pide confirmación ANTES de escribir (por si es el libro
      equivocado): en la individual con un diálogo; en lotes con un estado «ocupada»
      (Sobrescribir / Saltar) que dice de qué libro era.
    · RE-GRABAR una etiqueta con datos ANTIGUOS/desfasados: marca «Sobrescribir sin
      comprobar» en el diálogo de grabar. Entonces escribe en el PRIMER toque, SIN
      leer la etiqueta antes. Esto evita que Android lea el enlace viejo y abra una
      ficha inexistente en vez de dejar grabar (el hueco surgía porque la comprobación
      previa soltaba el control del NFC un instante; sin comprobación no hay hueco).
    · ANTI-DUPLICADO: una etiqueta física (su UID) solo puede estar en UN libro. Si
      la sobrescribes, se desvincula del libro antiguo y se avisa "reasignada
      (estaba en «X»)".
    · ETIQUETADO POR LOTES: selecciona libros en la Búsqueda y pulsa «📶 Etiquetar».
      Con 2+ libros pregunta EN QUÉ ORDEN etiquetar (la selección llega en orden
      arbitrario): 📥 por fecha de ingreso · 🗂️ por colección + nº de volumen · ↕️ como
      están seleccionados. Así la cola sigue un orden útil (y físico).
      Se abre una COLA que muestra cada ficha (portada, título, ubicación) y pide
      acercar la etiqueta; al grabar avanza. Mantiene el NFC en primer plano durante
      TODA la cola (un solo scan sostenido) y graba en el primer toque SIN leer la
      etiqueta antes: así re-graba etiquetas ya usadas sin que Android abra la ficha
      vieja ni entre libros (evita el redirect que rompía el flujo). MODO MANUAL
      (confirmar «Siguiente») o
      AUTOMÁTICO (avanza solo si la grabación es óptima). Pausar/Continuar, Saltar y
      Vaciar cola. La cola pendiente se guarda: si cierras, aparece «Reanudar
      etiquetado (N)». Truco: filtra antes «Etiqueta NFC = Sin» para etiquetar solo
      los que faltan.
    · Límite: requiere el móvil DESBLOQUEADO y la app/Chrome en primer plano (Web
      NFC no lee con la pantalla bloqueada).

  ── FUNCIONA SIN CONEXIÓN (PWA offline) ──────────────────────────────────────
    Tras instalarla y abrirla UNA vez con red, la app CARGA sin conexión (modo
    avión) para leer etiquetas NFC y ver la ubicación de un libro. Aparece un

  ── UBICACIONES (ámbitos y estanterías) ──────────────────────────────────────
    Gestiona la ubicación física como colecciones de estanterías: crear (incluso
    por lotes, p. ej. "Estanteria_pared_01..10"), renombrar, fusionar, mover,
    "explotar" (dejar sus libros sin ubicar), eliminar, añadir libros desde la
    búsqueda. Los desplegables de ámbito/estantería son <select> (fiables en móvil)
    y se reutilizan en el Inbox y en la edición de la ficha.
    · CADA FILA (ámbito/estante) muestra INLINE solo lo frecuente: el nombre (clic =
      ver sus libros en el Catálogo) y DOS iconos NFC DISTINTOS —📶 graba la etiqueta
      del PROPIO estante/ámbito; 🏷️ graba la NFC de CADA LIBRO que contiene (en lote,
      uno a uno, mismo flujo que «Etiquetar» del Catálogo)—. El resto de acciones
      (📋 ordenar · ➕ insertar · ✏️ renombrar · ➡️ mover · 🔀 fusionar · 🧹 explotar ·
      🗑 eliminar) van en un menú «⋯» para no llenar la fila de botones idénticos.
    · La asignación de estantería EN LOTE (📍 del Catálogo) IGNORA los DIGITALES (no
      van a un estante físico) y avisa cuántos saltó.
    · ESCANEAR la etiqueta NFC de una estantería/ámbito (o abrir su enlace ?amb=&est=)
      lleva al Catálogo FILTRADO por esa ubicación; si es una estantería concreta, ya
      ORDENADO por posición física (visible en «↕️ Ordenar», ajustable).
    · Los ámbitos salen COLAPSADOS por defecto y muestran cuántas estanterías y
      cuántos libros contienen. Clic en el NOMBRE de un ámbito/estantería = abre
      la Búsqueda filtrada por él (interactivo: seleccionar, etiquetar, mover…),
      sin tocar la selección que ya tuvieras acumulada.
    · 📋 ORDENAR ESTANTERÍA (por su posición física, para localizar libros e
      inventariar): botón 📋 en cada estantería (y en el chip de filtro cuando ves
      una balda en la Búsqueda). Abre la estantería como LISTA vertical donde
      colocas los libros en su orden real:
        - ARRASTRANDO las filas (escritorio) o con ↑/↓ (móvil), o
        - 📶 ORDENANDO ESCANEANDO: escaneo NFC continuo; tocas los libros en su
          orden físico y se numeran solos 1,2,3… Detecta y avisa (con vibración
          distinta): doble ACCIDENTAL (mismo tag < 5 s → se ignora), libro AJENO
          a la balda, y repetido DELIBERADO (ya escaneado, no se recoloca).
      Al ver una estantería, sus libros salen en ese orden y cada portada muestra
      su nº de posición (#N). "Guardar orden" lo fija. Los libros aún SIN posición se
      presentan por ORDEN DE INGRESO (no por título), así el orden inicial coincide
      con el de ingesta; al «Guardar orden» ese queda fijado.
    · Acciones en LOTE en la Búsqueda (barra de selección): además de mover a una
      estantería 📍, ahora «🚫 Quitar de estantería» (pasan a "Sin asignar", sin
      borrar nada) y «📶 Seleccionar por NFC» (toca etiquetas para ir sumándolos a
      la selección).

  ── AUTORES ──────────────────────────────────────────────────────────────────
    Página para consultar y DEPURAR autores. Buscador por nombre o grafía
    alternativa; sin búsqueda, salen los autores CON libros, más usados primero
    (los miles de nombres del volcado sin libros aparecen al buscarlos). Cada
    tarjeta muestra foto, nombre, «a.k.a.» (variantes), nº de libros y fechas.
    · Filtros: Foto (con/sin), Biografía (con/sin), ROL (autor/traductor/ilustrador/
      prologuista/anotador/editor/compilador), MÍN. OBRAS (≥1/2/3/5/10, por el rol
      filtrado) y Orden (nº de obras / nombre). El filtro de rol lista a quienes
      intervienen en ese rol (nº = libros en ese rol).
    · ELIMINAR (admin): en Modo selección, «🗑 Eliminar N» borra los autores marcados
      pero SOLO los que no figuran en NINGÚN documento (ni autor ni contribuyente); los
      referenciados se conservan (salvaguarda anti-pérdida). En la ficha de un autor SIN
      obras aparece «🗑 Eliminar autor». (POST /autores/eliminar {ids}.)
    · Clic en una tarjeta → FICHA del autor: foto, biografía, fechas, «también
      conocido como», sus ROLES y sus libros. Los libros EN PAPEL se listan primero
      (subtítulo «📖 En papel») y los ELECTRÓNICOS después; los que ya tienen etiqueta
      NFC llevan un badge 📶. El admin puede EDITAR esos datos y SUBIR una foto; los
      botones (compactos, para no distanciar de la lista de libros) Guardar/Cerrar están
      DUPLICADOS arriba (bajo la foto, junto a Autocompletar) para no bajar hasta el pie.
      «🖼️ De obra» fija como foto una imagen (portada o interior) de uno de sus libros
      —combinado con «🖹 Del documento» de la ficha del libro, permite usar la foto del
      autor del interior del libro. (GET /autores/:id/imagenes-obras.)
    · MOSTRAR LIBROS EN EL CATÁLOGO (desde la ficha): «🔍 Ver en Catálogo» abre el
      Catálogo con TODOS sus libros (como autor o contribuyente). Para mandar solo
      ALGUNOS: «🖱 Modo selección» (tocar una tarjeta la marca, ✓, en vez de abrir su
      ficha) → «🔍 Mostrar en Catálogo» los manda YA SELECCIONADOS (llegan en modo
      selección, listos para actuar). (Debajo, /catalogo acepta ?autor= —autor o
      contribuyente— y ?ids= —CSV de libros—.)
    · DEPURAR AUTORES (admin, desde la ficha):
      · 🔀 FUSIONAR OTRO AQUÍ: busca otro autor y lo ABSORBE en este (sus libros pasan
        aquí; el otro se borra). Para unificar duplicados. (POST /autores/fusionar
        {destino, ids}; resuelve por resolverPersona insensible a grafía.)
      · 🚫 QUITAR AUTORÍA: quita este autor de TODOS sus libros, o solo de los
        SELECCIONADOS (en modo selección). Los documentos quedan SIN ese autor; si se
        queda sin obras, el autor se BORRA (nunca se borra un autor con obras). Ideal
        para REVISTAS/anónimos (recomendado: dejarlas SIN autor — su identidad es la
        cabecera/ISSN, no una persona) o para deshacer una autoría errónea.
        (POST /autores/:id/quitar {ids?}.)
    · ROLES de contribuyente: la ficha del LIBRO muestra «Contribuciones» (traductor:
      X · ilustrador: Y) e «Idioma original». Se capturan en la ingesta desde
      OpenLibrary (mención `by_statement`) + la MENCIÓN de la BNE del Fichero (roles y
      abreviaturas: trad./ilustr./pról./introd./pref.) + la lengua original del Fichero,
      SIN IA. OJO: los libros ya catalogados NO tienen roles hasta correr el backfill
      (por ISBN, tandas de 25, reanudable):
        docker exec gestor-biblioteca node scripts/roles-autores.js --ejecutar
      (sin --ejecutar cuenta; --limite N para probar).
    · COMBINAR (admin): activa «🖱 Modo selección» (o pulsación larga / doble clic en
      una tarjeta) — MISMA mecánica y pista visual que el Catálogo/estantes: cada
      tarjeta muestra su círculo de selección y la marcada se «hunde». Marca 2+
      autores → «🔗 Combinar» → elige el que
      se CONSERVA (destino B; por defecto el que más libros tiene). Los demás (A)
      se funden en B: sus nombres pasan a «también conocido como» de B y TODOS sus
      libros se reasignan a B; luego se borran las A. Sin pérdida (los libros y sus
      ficheros no se tocan). Es la versión interactiva de scripts/backfill-autores.
    · ✨ AUTOCOMPLETAR (web) (admin, en la ficha): rellena foto, biografía,
      seudónimos/heterónimos y fechas desde OpenLibrary + Wikidata + Wikipedia
      (gratis, sin clave y SIN IA). Conservador: solo rellena lo que falte. Para
      rellenar TODO el catálogo de golpe, por TANDAS DE 25:
        docker exec gestor-biblioteca node scripts/enriquecer-autores.js --ejecutar
      (sin --ejecutar = cuenta cuántos faltan; --limite N para probar; es
      reanudable: cada pasada coge los que aún no tienen bio/foto).

  ── EDITORIALES ──────────────────────────────────────────────────────────────
    Página GEMELA de Autores para consultar y DEPURAR editoriales. Buscador por
    nombre o grafía alternativa; sin búsqueda, salen las editoriales CON libros,
    las más usadas primero (orden por nº de libros o por nombre). Cada tarjeta:
    nombre + nº de libros (+ «a.k.a.» si tiene nombres alternativos).
    La lista está PAGINADA (60/pág, como Autores): recuento «desde–hasta de TOTAL»
    y controles ‹ ⏮ ‹ N/M › ⏭ ›. Buscar o cambiar el orden vuelve a la página 1.
    · FICHA (clic en una tarjeta): nombre, «también conocida como», LOGO (subir o
      elegir de un libro), sede (ciudad/país), años de actividad e historia —
      EDITABLES (admin). Y los LIBROS que publica (clic → ficha del libro). «🔍 Ver
      en Catálogo» abre todos sus libros filtrados (GET /catalogo?editorial=<id>).
    · SELECCIÓN de editoriales (lista): MISMA mecánica y pista visual que el resto
      (Catálogo/Autores) — «🖱 Modo selección» (botón o pulsación larga / doble clic
      en una tarjeta), círculo de selección, la marcada se «hunde».
    · SELECCIÓN de LIBROS dentro de la ficha (admin): «🖱 Modo selección» sobre los
      libros de la editorial y actuar sobre los marcados:
        – 🔍 Mostrar en Catálogo (llegan ya seleccionados);
        – ➡️ A otra editorial — reasigna los seleccionados (POST /editoriales/:id/
          reasignar); esta editorial conserva los no marcados (y si queda vacía, se borra);
        – 🚫 Quitar editorial — los seleccionados quedan SIN editorial (POST
          /editoriales/:id/quitar);
        – 🏢 Reclasificar — ver «Reclasificar editorial» abajo.
    · COMBINAR (admin): marca 2+ editoriales → «🔗 Combinar» → elige la que se
      CONSERVA (B; por defecto la de más libros). Las demás (A) se funden en B:
      sus nombres pasan a «también conocida como» de B y TODOS sus libros se
      reasignan a B (biblioteca.editorial); luego se borran las A. Sin pérdida
      (los libros y sus ficheros no se tocan). En la ficha: «🔀 Fusionar esta en…»
      (elige la destino en un overlay). (POST /editoriales/fusionar.)
    · 💥 EXPLOTAR (admin, en la ficha): LIBERA todos sus libros (quedan sin
      editorial) y BORRA la editorial. Distinto de fusionar (que los reasigna a
      otra) y de borrar (que solo permite las vacías). No se pierde ningún libro
      (solo se les quita la referencia; se pueden reclasificar después). (POST
      /editoriales/:id/explotar.)
    · BORRAR (admin, en la ficha): solo una editorial SIN libros (fantasma tras
      una fusión/corrección; nunca se borra una con obras). (POST /editoriales/
      :id/borrar.) Backend: src/utils/gestion-editoriales.js.

    ── RECLASIFICAR EDITORIAL (de una selección de libros) ──
    Arregla en lote el campo `editorial` de una SELECCIÓN de libros (editoriales
    mal puestas —p. ej. «ePubLibre», que es un grupo de maquetación, no una casa—
    o ausentes). Se lanza desde:
      · el CATÁLOGO — selección de libros → «🏢 Reclasificar editorial»;
      · la ficha de una editorial — libros seleccionados → «🏢 Reclasificar».
    Busca la editorial correcta de cada libro EN CASCADA, de lo mas BARATO a lo mas
    caro (minimiza IA/coste):
      0) EDITORIAL en el nombre de archivo (corchete «[Catedra] …» que puso el
         curador) — GRATIS.
      1) Fichero local (por ISBN, offline).
      2) COLECCION -> editorial (GRATIS): muchas colecciones son marca de UNA casa
         («Biblioteca Clasica Gredos»->Gredos, «Ancora y Delfin»->Destino, «Letras
         Hispanicas»->Catedra, «Tus Libros»->Anaya…). Mapa determinista en
         src/utils/coleccion-editorial.js. Clave para los libros SIN editorial (ebooks
         sin ISBN): el dato ya esta guardado, sin abrir ficheros ni vision.
      3) OpenLibrary  4) Google Books.
      5) COLECCION -> IA de TEXTO (barata, SIN vision) — OPT-IN: el modelo suele
         conocer la casa de una coleccion por su nombre. Antes de la vision.
      6) IA (VISION sobre la portada) — OPT-IN y ULTIMO recurso (la portada NO muestra
         la coleccion/colofon, asi que aqui rara vez aporta la editorial; lenta y con
         coste). Las plataformas de autopublicacion (CreateSpace/KDP/Lulu) y los
         repackagers (ePubLibre/DigiCat…) NO cuentan como editorial (utils/editoriales-falsas.js).
    Decisión por libro (anti-pérdida): si halla una editorial real distinta →
    propone el cambio; si no halla nada y la actual es FALSA (ePubLibre…) → propone
    QUITARLA (queda sin editorial); si no halla nada y la actual es real → la DEJA.
    Presenta un INFORME por TRANSICIÓN (no un log por libro): «N libros de «Y» a
    «Z» [fuente]», «N quitados de «M» (sin editorial)», + sin cambio / no resueltos.
    DRY-RUN primero (previsualiza sin tocar nada); «✅ Aplicar cambios» ejecuta lo
    previsualizado (resuelve/crea la editorial destino por nombre). Proceso en 2º
    plano con progreso (POST /editoriales/reclasificar[/aplicar], GET …/estado).
    CLI en lote (NAS): node scripts/reclasificar-editoriales.js --editorial "ePubLibre"
    [--ejecutar] [--ia] (dry-run por defecto; también --id/--sin-editorial/--todos).
    Backend: src/utils/reclasificar-editorial.js.

  ── ACTIVIDAD (administración) ───────────────────────────────────────────────
    Fichas plegables: 👁 Vigilante del Inbox (activar/pausar), 🧭 Estado del
    Conformador, 🧹 Lanzar mantenimiento, 🎯 Campañas de fondo (ver abajo), 🩺
    Integridad (diagnosticar/reparar a la Papelera), 🧼 Sanear catálogo, 👁️ Visión
    IA (proveedores/claves), 🔎 Búsqueda (reindexar FTS), 🔞 Permisos de invitados
    (permitir/restringir NSFW), 📜 Logs en vivo.

    · 🎯 CAMPAÑAS DE FONDO (admin): rellenan información INCOMPLETA del catálogo
      cuando el sistema está en REPOSO (Inbox inactivo), cada una con su interruptor,
      su tamaño de LOTE y su periodicidad (cada N minutos). Comparten el candado con
      la ingesta y le CEDEN el turno. Cada campaña avisa de su COSTE:
        🆓 sin IA (local/Fichero) · 🌐 APIs gratis (con LÍMITE de llamadas) · 🤖 IA de pago.
      Campañas actuales:
        · 🆓 Roles e idioma original — traductor/ilustrador/… e idioma original por
          ISBN (mención de la BNE del Fichero + OpenLibrary).
        · 🆓 Título e idioma original (del fichero) — lee la PÁGINA DE CRÉDITOS del
          EPUB/PDF y recupera el «Título original» (y todos, en antologías) + un indicio
          de idioma original, SIN IA (utils/titulo-original.js, el mismo parser que el
          backfill scripts/recuperar-titulo-original.js). Título e idioma original se
          intentan SIEMPRE, de cualquier fuente (APIs + este fichero); solo se guardan/
          muestran si DIFIEREN del título/idioma del propio documento. Ambos salen en la
          ficha (Datos catalográficos).
        · 🌐 Huecos de metadatos — sinopsis, año, editorial, autores, palabras clave
          y Dewey/LCC por ISBN (OpenLibrary/Google Books/DNB). Conservador.
        · 🌐 Fotos y biografías de autor — OpenLibrary + Wikidata + Wikipedia.
        · 🤖 Completar a fondo (leer el libro) — lee las PÁGINAS del propio fichero
          (portadilla/contraportada/créditos) con la visión y una plantilla rica, y
          completa autores/roles reales, sinopsis e identificadores que las APIs no
          tienen. Se centra en libros con el autor puesto a la EDITORIAL (DK, VV.AA.)
          o sin autor, y solo aplica si la extracción MERECE LA PENA (evaluador de
          calidad). Consume visión (gratis→pago).
        · 🤖 Descripciones CDU/Dewey/LCC — se generan POR LOTE (varios códigos por
          llamada de IA → menos coste; DESC_LOTE_IA=6), con fallback uno-a-uno.
      Cada campaña es RESUMIBLE y TERMINABLE: sella cada elemento procesado (con o sin
      datos) para no repetirlo; el contador «Pendientes» baja hasta 0. Arrancan todas
      DESACTIVADAS (opt-in). «▶ Ahora» lanza una tanda inmediata en 2º plano y muestra
      una BARRA DE PROGRESO bajo la campaña mientras corre. «⏩ Completar» hace un
      BACKFILL COMPLETO: encadena tandas hasta dejar 0 pendientes (en 2º plano, cediendo
      a la ingesta); «⏹ Detener» lo para. Solo un backfill a la vez. Si al pulsar «Ahora» sale
      «Ocupado: … en curso», es que el candado ÚNICO lo tiene otro proceso (ingesta /
      mantenimiento / otra campaña): solo corre un trabajo pesado a la vez. Solo corren
      en el NAS (contenedor). Config persistida en Mongo (ajustes/campanas).
      · IA de TEXTO (clasificación CDU + descripciones) usa la MISMA rotación
        multi-proveedor que la visión: Gemini free → Groq/OpenRouter free → Gemini de
        PAGO como último recurso (gestionada en «👁️ Visión IA»).

  ── CUARENTENA ───────────────────────────────────────────────────────────────
    Lo que no se pudo catalogar, por categorías plegables:
      · no-identificados (sin título tras todos los intentos)
      · ilegibles (PDF/EPUB corrupto o 0 bytes): flujo de arreglo — buscar una
        copia sana (Anna's/Z-Lib/Libgen/Gutenberg), SUBIRLA y "Procesar".
      · duplicados (mismo identificador, distinto contenido): dedup manual.
    Cualquier depósito se puede DESCARTAR (va a la Papelera, recuperable), por ítem
    o por categoría entera (re-pide contraseña).

  ── PAPELERA (Recycling) ─────────────────────────────────────────────────────
    Todo lo que se «borra» (ficheros reemplazados, carpetas de documentos eliminados,
    depósitos descartados…) se MUEVE aquí en subcarpetas serializadas, nunca se elimina
    de verdad (política anti-pérdida). Tabla con tamaño/ficheros por subcarpeta + «ver».
    · ↩️ RESTAURAR (admin): devuelve el fichero/carpeta a su UBICACIÓN ORIGINAL, como la
      Papelera de Windows. Funciona con un MANIFIESTO (.papelera.json) que registra el
      origen al reciclar; NUNCA pisa lo que ya exista en el destino (esa parte se conserva
      en la Papelera y se informa como conflicto). Las entradas recicladas ANTES de esta
      función no tienen manifiesto → no ofrecen «restaurar». (POST /papelera/restaurar
      {sub}. papelera.js·restaurar.)
    · 🗑 Vaciar (subcarpeta o toda): borrado DEFINITIVO explícito del usuario.


────────────────────────────────────────────────────────────────────────────────
8. API REST — REFERENCIA DE ENDPOINTS
────────────────────────────────────────────────────────────────────────────────

  Base URL: http://ip-del-nas:3000  (o http://localhost:3000 en local)

  ──────────────────────────────────────────────────────────────────────────────
  GET /health
  ──────────────────────────────────────────────────────────────────────────────
  Comprueba que la app está en marcha.

  Respuesta 200:
    { "status": "ok" }

  ──────────────────────────────────────────────────────────────────────────────
  GET /recursos/<ruta>
  ──────────────────────────────────────────────────────────────────────────────
  Sirve ficheros del árbol CDU (portadas, imágenes, EPUBs, PDFs…).
  La ruta relativa coincide con el campo "portada" o "imagenes[].ruta" del
  documento en MongoDB.

  Ejemplo:
    GET /recursos/141.78/libros/978-84-1234-567-8/portada.jpg

  ──────────────────────────────────────────────────────────────────────────────
  POST /api/ingestar
  ──────────────────────────────────────────────────────────────────────────────
  Ingesta uno o varios ficheros enviados directamente (sin pasar por el Inbox).
  Útil para integración con otras apps o para subidas manuales puntuales.

  Tipo de petición: multipart/form-data
  Campo de archivo: "files"  (uno o varios)
  Campos opcionales (form fields):
    ubicacion    JSON con { "ambito": "...", "estanteria": "..." }
                 Indica la ubicación física del libro (para libros en papel).
    ambito       Alternativa a ubicacion (junto con estanteria)
    estanteria   Alternativa a ubicacion (junto con ambito)

  Ejemplo con curl:
    curl -X POST http://localhost:3000/api/ingestar \
         -F "files=@/ruta/al/libro.epub" \
         -F 'ubicacion={"ambito":"Salón","estanteria":"A-3"}'

  Respuesta 200 (todo OK):
    {
      "status": "success",
      "resultados": [
        {
          "ok": true,
          "operacion": "insercion"|"actualizacion",
          "estado": "completado"|"pendiente",
          "id": "objectid",
          "isbn": "978-...",
          "titulo": "El título del libro",
          "ruta": "/recursos/..."
        }
      ]
    }

  Respuesta 207 (parcial — algunos fallaron):
    {
      "status": "partial",
      "resultados": [
        { "ok": true,  ... },
        { "ok": false, "destino": "cuarentena"|"reintentos", "error": "..." }
      ]
    }

  ──────────────────────────────────────────────────────────────────────────────
  POST /api/mantenimiento
  ──────────────────────────────────────────────────────────────────────────────
  Dispara el Conformador MANUALMENTE. El mantenimiento NO corre solo al quedar el
  Inbox inactivo (salvo opt-in MANTENIMIENTO_ACTIVO=1): se lanza desde aquí.
  Vacía todo el backlog de documentos pendientes de conformar, en RONDAS de
  MANTENIMIENTO_LOTE (25) documentos, en segundo plano. Cede a la ingesta del
  Inbox entre rondas y REINTENTA hasta vaciar el backlog (no se abandona a medias).
  El progreso se sigue en los logs del contenedor.

  Parámetros (en query o body):
    activar  (M):  CUÁNDO arrancar.  0 = ya (defecto) · N>0 = en N segundos ·
                   -1 = cuando el Inbox quede inactivo.
    intervalo (N): pausa en segundos entre una ronda de 25 y la siguiente.
                   0 (defecto) = continuo, sin pausa. Útil para correr suave y
                   desatendido sin saturar Atlas/APIs.
    POST /api/mantenimiento?activar=-1&intervalo=30
    POST /api/mantenimiento     body { "activar": 300, "intervalo": 0 }

  Para DETENER un mantenimiento manual en marcha (o cancelar uno programado):
    POST /api/mantenimiento/modo   body { "modo": "apagado" }

  Respuesta 202 (programado):
    { "ok": true, "mensaje": "Mantenimiento programado (cuando el Inbox quede inactivo; 30s entre rondas de 25). Detén con modo=apagado." }

  Respuesta 409 (ya hay un mantenimiento manual en curso):
    { "ok": false, "motivo": "ya hay un mantenimiento manual en curso" }

  ──────────────────────────────────────────────────────────────────────────────
  GET /api/mantenimiento/estado
  ──────────────────────────────────────────────────────────────────────────────
  Devuelve el estado actual del Conformador.

  Respuesta 200:
    {
      "modo":          "diferido"|"apagado"|"apagado-hasta",
      "dormido":       true|false,
      "apagadoHasta":  "2026-06-20T00:00:00.000Z"|null,
      "ultimaRevision":"2026-06-19T14:32:10.000Z"|null
    }

  Campos:
    modo          Modo de funcionamiento actual (ver sección 8)
    dormido       true = la cola está vacía; el Conformador no consulta la BD
                  hasta la próxima ingesta
    apagadoHasta  ISO 8601 del momento en que expira el apagado temporal
                  (solo presente en modo "apagado-hasta")
    ultimaRevision ISO 8601 de la última pasada completada (null si nunca corrió)

  ──────────────────────────────────────────────────────────────────────────────
  POST /api/mantenimiento/modo
  ──────────────────────────────────────────────────────────────────────────────
  Cambia el modo del Conformador en caliente (sin reiniciar la app).

  Body: application/json
    {
      "modo":  "diferido"|"apagado"|"apagado-hasta",
      "hasta": "proxima-hora"|"proximo-dia"|"proxima-semana"
    }

  "hasta" solo es necesario cuando modo="apagado-hasta".

  Modos disponibles:
  ─────────────────
  "diferido"        Modo por defecto. El Conformador corre automáticamente
                    tras MANTENIMIENTO_REPOSO_MS de Inbox inactivo.
                    Si estaba apagado, lo reactiva.

  "apagado"         Desactiva el Conformador indefinidamente.
                    Solo vuelve a correr si se cambia el modo manualmente
                    o se llama a POST /api/mantenimiento.

  "apagado-hasta"   Desactiva el Conformador hasta el hito indicado en "hasta",
                    tras el cual vuelve automáticamente a modo "diferido".

  Valores de "hasta":
  ───────────────────
  "proxima-hora"    Hasta el inicio de la próxima hora en punto.
                    (Ej.: si son las 14:37, hasta las 15:00)
  "proximo-dia"     Hasta mañana a las 00:00.
  "proxima-semana"  Hasta dentro de 7 días a las 00:00.

  Para disparo INMEDIATO: usar POST /api/mantenimiento (ver arriba).

  Ejemplos con curl:

    # Desactivar el Conformador hasta mañana:
    curl -X POST http://localhost:3000/api/mantenimiento/modo \
         -H "Content-Type: application/json" \
         -d '{"modo":"apagado-hasta","hasta":"proximo-dia"}'

    # Volver al modo automático:
    curl -X POST http://localhost:3000/api/mantenimiento/modo \
         -H "Content-Type: application/json" \
         -d '{"modo":"diferido"}'

    # Apagar indefinidamente:
    curl -X POST http://localhost:3000/api/mantenimiento/modo \
         -H "Content-Type: application/json" \
         -d '{"modo":"apagado"}'

  Respuesta 200 (OK):
    {
      "ok": true,
      "modo": "apagado-hasta",
      "dormido": false,
      "apagadoHasta": "2026-06-20T00:00:00.000Z",
      "ultimaRevision": "..."
    }

  Respuesta 400 (parámetros inválidos):
    { "ok": false, "motivo": "Modo inválido: \"xyz\". Valores: diferido, apagado, apagado-hasta." }

  ──────────────────────────────────────────────────────────────────────────────
  GET /api/estadisticas
  ──────────────────────────────────────────────────────────────────────────────
  Estadísticas del catálogo (para un panel/front-end).

  Query opcional:
    detalle=0   Devuelve solo los recuentos (omite las listas largas de revistas
                y de CDU). Útil cuando el catálogo es grande.

  Respuesta 200:
    {
      "generado": "2026-06-22T05:33:13.579Z",
      "total": 4858,                 // documentos totales
      "libros": 4762,
      "revistas_total": 96,          // nº de números de revista catalogados
      "revistas": {

9. EL CONFORMADOR — PROCESO DE MANTENIMIENTO
────────────────────────────────────────────────────────────────────────────────

  El Conformador es un proceso durmiente que revisa periódicamente los
  documentos de la base de datos y les aplica "tareas de mantenimiento"
  (completar datos, mejorar portadas, regenerar sidecars…).

  Comportamiento por defecto:
    · Espera a que el Inbox lleve inactivo ≥ MANTENIMIENTO_REPOSO_MS (5 min)
    · Procesa hasta MANTENIMIENTO_LOTE (25) documentos por pasada
    · Cede el turno a la ingesta si llega algo al Inbox
    · Cuando la cola queda vacía, entra en "reposo" (no vuelve a consultar
      la BD hasta que se ingeste un nuevo documento)
    · Solo corre dentro del contenedor Docker (donde están los ficheros);
      un arranque local no "conforma" documentos del NAS por error

  Tareas actuales (en src/mantenimiento/tareas.js), EN ORDEN (la cadena importa):
    restaurar-original         Repone el .epub/.pdf si falta en la carpeta, desde
                               Reintentos/Cuarentena
    re-enriquecer-degradados   Docs con ISBN pero metadata pobre (lote con APIs
                               caídas): re-busca por ISBN y corrige título/autor
    re-clasificar-cdu          Re-clasifica la CDU (BNE→DNB→caché→IA) y mueve la
                               carpeta al árbol nuevo si cambió
    describir-cdu              Asegura la descripción bilingüe (ES/EN) del CDU en
                               la colección cdu_descripciones
    completar-hash-contenido   Calcula el SHA-256 del fichero (identidad 1:1)
    completar-coleccion        Detecta la colección/volumen del nombre de archivo
    completar-nombre-archivo   Rellena nombre_archivo si falta
    revisar-portada            Mejora portadas de baja calidad o ausentes
                               (no toca libros en papel/escaneados)
    generar-sidecars-pdf       Rasteriza páginas clave como sidecars de imagen
    regenerar-registros        Reescribe registro.json y registro.marc.xml desde
                               la BD (autores/editorial por nombre)

  Firmas: cada tarea tiene una versión; subir la versión fuerza que todos
  los documentos vuelvan a pasar por esa tarea en la próxima pasada.

  Por defecto el mantenimiento es MANUAL (no corre solo); se dispara con
  POST /api/mantenimiento (activar/intervalo). Modos del AUTO (opt-in), cambiables
  en caliente vía API:
    diferido      Auto, tras MANTENIMIENTO_REPOSO_MS de inactividad (opt-in)
    apagado       No corre solo (DEFECTO); además DETIENE un manual en curso
    apagado-hasta Pausa temporal del auto hasta una hora/día/semana

10. GESTIÓN DE FALLOS
────────────────────────────────────────────────────────────────────────────────

  9.1  Cuarentena
       Destino: /volume3/BIBLIOTECA DIGITAL/Cuarentena/

       Ficheros que no se pudieron catalogar. Se organizan en SUBCARPETAS por
       categoría (cada depósito lleva su estado.json con el error):
         duplicados        Mismo identificador (ISBN/hash) que algo ya catalogado
                           pero CONTENIDO DISTINTO (otra edición/versión): a revisar.
                           OJO: una copia EXACTA (mismo hash de contenido) ya NO llega
                           aquí — se confirma por hash y se manda a la Papelera (no
                           satura Cuarentena). Solo lo que difiere queda para revisar.
         no-identificados  Sin título/ISBN tras agotar texto/APIs/OCR/IA.
         ilegibles         Fichero DEFECTUOSO (p. ej. PDF con el xref dañado /
                           sin páginas, EPUB con el ZIP roto). Necesita una
                           copia mejor. Se detecta en el primer paso y NO se
                           cataloga (no deja registro en la BD).
         otros             Cualquier otro error.

       Acción: revisar; para 'ilegibles' conseguir una copia buena y soltarla en
       el Inbox; para 'no-identificados' renombrar con título/autor/ISBN si se
       puede y reintentar; 'duplicados' se pueden borrar (ya están en el catálogo).

  9.2  Reintentos
       Destino: /volume3/BIBLIOTECA DIGITAL/Reintentos/

       Contiene documentos que fallaron por causas de infraestructura
       (red caída, MongoDB Atlas inaccesible…). Se reintentarán automáticamente
       en la próxima pasada del Conformador.

       Cada entrada tiene:
         · El fichero original (o copia íntegra)
         · estado.json con el error y los datos parciales

13. NOTAS IMPORTANTES SOBRE EL NAS SYNOLOGY
────────────────────────────────────────────────────────────────────────────────

  · El NAS usa Compose v1: el comando es "docker-compose" (con guion), no
    "docker compose".

  · Los eventos de fichero de chokidar NO se propagan por el bind mount
    al soltar ficheros por SMB. La app lo compensa con un escaneo periódico
    (VIGILANTE_ESCANEO_MS). Por eso no pongas VIGILANTE_POLLING=0 en el NAS.

  · La carpeta @eaDir que crea Synology dentro del Inbox (miniaturas/metadatos)
    es ignorada automáticamente por el vigilante.

  · El contenedor corre como root (necesario para leer/escribir los ficheros
    aunque SMB los entregue con permisos restrictivos).

  · El volumen anónimo de node_modules persiste entre reinicios pero se
    elimina con "down -v". Tras cambiar dependencias en package.json, siempre
    usar "down -v" antes de "up --build".

  ── SEGURIDAD AL COMBINAR AUTORES / EDITORIALES ──────────────────────────────
    La fusión BORRA las absorbidas y es IRREVERSIBLE. BUG corregido (2026-07-10): el
    diálogo listaba solo las entidades VISIBLES en la búsqueda/página actual, pero enviaba
    TODA la selección —que sobrevive a búsquedas y a la paginación—, así que podía absorber
    entidades que el usuario no veía. (Caso real: al fusionar «Geoplaneta»+«GeoPlaneta» se
    tragó además «Seix Barral», con 292 libros.) Ahora una CACHÉ de lo marcado
    (_autoresSelInfo / _editorialesSelInfo, id → nombre + nº de libros) hace que el diálogo
    liste EXACTAMENTE lo que se va a fusionar, aunque ya no esté en pantalla; y si se van a
    reasignar más de 25 libros hay que marcar una casilla de confirmación explícita.

  -- EDITORIALES: datos de la ficha --------------------------------------------
    Ademas de nombre y "tambien conocida como", la editorial guarda (campos ADITIVOS: la
    coleccion `editoriales` no tiene validador $jsonSchema, no hace falta setup-mongo):
      * logo / logos[]      -- LOGO. Se SUBE (boton 'Logo') o se ELIGE de una imagen de uno
                               de sus libros (boton 'De un libro': portadas + carrusel). El
                               anterior no se borra: se acumula en logos[] y solo cambia el
                               principal. Se guarda en CDU/_editoriales/<id>/ y se sirve por
                               /recursos (como la foto de autor). Se pinta con object-fit:
                               contain (un logo no se recorta) en la ficha y en la tarjeta.
      * descripcion         -- historia / notas libres.
      * fecha_fundacion     -- ano (1000-2100). Vacio = sin dato.
      * fecha_disolucion    -- ano. Vacio = sigue activa.
      * ciudad / pais       -- sede.
    Endpoints: POST /editoriales/:id/logo (base64, solo admin) y
    GET /editoriales/:id/imagenes-libros (imagenes de sus libros para elegir el logo).
    La validacion de imagen (utils/imagen-base64.js, compartida con la foto de autor) exige
    MAGIC BYTES reales (jpg/png/webp): antes un base64 de texto cualquiera se escribia como
    un .jpg corrupto.

  -- INGESTA: colaboradores, editorial real y titulo original --------------------
    Tres mejoras en la extraccion (la mayoria en la PRIMERA llamada a la IA, sin coste extra):
      * COLABORADORES separados: una mencion con VARIOS ilustradores/traductores unidos por
        « & »/« ; »/« / » (p. ej. "Rochegrosse & Rackham & Clarke (Ilustrador)") se reparte
        ahora en UNA persona por cada nombre (antes se guardaba como un unico "autor" fusionado).
        El split (utils/autor-normalizar.js · separarAutores, que respeta "Ortega y Gasset")
        se aplica en motor-catalogo y en el Conformador (enriquecer-a-fondo). Para lo YA
        catalogado hay backfill: scripts/separar-autores-fusionados.js (dry-run por defecto).
      * EDITORIAL REAL desde el principio: los maquetadores/re-editores de ebooks (ePubLibre,
        Lectulandia, DigiCat, Good Press, epubGratis, e-artnow, Musaicum) y las plataformas de
        AUTOPUBLICACION (CreateSpace/KDP/Lulu) NO son editoriales (lista unica en
        utils/editoriales-falsas.js). El motor de enriquecimiento deduce la casa REAL asi:
        (1) MAPA determinista COLECCION->editorial, GRATIS ("Biblioteca Clasica Gredos"->Gredos,
        "Ancora y Delfin"->Destino, "Letras Hispanicas"->Catedra, "Tus Libros"->Anaya…, en
        utils/coleccion-editorial.js) — gana a las APIs cuando el archivo trae un maquetador o
        no hay editorial; (2) la IA de vision recibe instruccion de leer la casa del colofon/
        creditos o de la coleccion e ignorar el maquetador. Una editorial de las APIs que sea a
        su vez un repackager (DigiCat…) se descarta, para que nunca gane (era el caso original).
        El mismo mapa lo usa el reclasificador de editorial en lote (ver su seccion).
      * TITULO ORIGINAL (obras traducidas): nuevo campo `titulo_original` (y `titulos_originales`
        para antologias de relatos, cuyos originales figuran en la pagina de creditos). Lo pide
        la plantilla de vision y lo rellena tambien el Conformador. Campos ADITIVOS (biblioteca
        es additionalProperties-abierto: sin setup-mongo). Backfill para lo ya catalogado:
        scripts/recuperar-titulo-original.js (lee la pagina de copyright del EPUB/PDF, SIN IA;
        dry-run por defecto).

  -- FICHA: previsualizar MOBI/AZW3 · autores por rol · titulos artefacto ----------
    * MOBI/AZW3 se PREVISUALIZAN en la ficha («📖 Leer»): el backend extrae el TEXTO
      (/documentos/:id/pagina-texto, conservando estructura) y el front lo renderiza en
      un iframe SANDBOX (sin scripts, aislado). Con DRM o compresion no soportada
      (HUFF/CDIC) → aviso + descarga. (No hay lector nativo de mobi en el navegador como
      para EPUB/PDF; esto es una previsualizacion del texto, no un lector paginado.)
    * Ficha de AUTOR: junto a «Libros (N)» se muestra el DESGLOSE por rol —siempre
      «autor N» (aunque sea 0) + cada funcion con libros (ilustrador/traductor/…)—, para
      que un ilustrador no parezca «autor de 0 libros». (El recuento de la LISTA de
      autores sigue siendo por rol AUTOR; filtra por rol para ver los de contribucion.)
    * TITULOS ARTEFACTO: esTituloArtefacto (utils/parsear-nombre.js) detecta mas casos —
      «New Page N», cabeceras de correo («Subject:/From:…»), marcas de scene («TeAM
      YYePG», «ebooks from…») y nombres de fichero fuente con extension separada («Hawking
      FM» = FrameMaker). Si tras enriquecer el titulo SIGUE siendo artefacto (ninguna
      autoridad lo sustituyo), el doc NO se da por completado: se marca `pendiente` +
      `revision_requerida` + `titulo_artefacto` (para relanzar una investigacion mas
      profunda —vision del libro— en el Conformador / revision en el panel).
